\begin{abstract}
On CPU endpoints, the GPU heuristic---higher bit-width yields higher throughput---is wrong.
We present a mechanistic, cross-platform study of GGUF K-quant inference on ARM (Pixel~6a)
and x86 (Intel i5-1235U), finding three non-obvious results.
(1)~Throughput is non-monotonic: Q2\_K leads at 7.49~tok/s (ARM) and 14.1~tok/s (x86),
while Q6\_K is \emph{slowest} (3.53 and 6.8~tok/s)---a $2.1\times$ intra-format spread.
The root cause is SIMD dequantization overhead: Q6\_K's split-bit layout requires
$\approx$26 NEON ops/block versus $\approx$10 for Q2\_K (from static analysis of
llama.cpp commit \texttt{1a29907}; see Table~\ref{tab:superblock}).
(2)~Apple M4 Metal reverses the ordering entirely (Q4\_K\_S fastest, Q8\_0 slowest),
confirming the non-monotonicity is a CPU property; M4 CPU-only beats Metal for Q8\_0
($1.97\times$).
(3)~Q2\_K exhibits a KV-cache context cliff ($-48\%$ ARM at ctx=512; $-50\%$ x86 at
ctx=1300--1400), predicted within 5\% by $L2_\text{cache}/1024$ on both platforms.
Results validated on 1,200+ measurements across Llama~3.2~3B and Qwen~2.5~1.5B.
\end{abstract}

\begin{IEEEkeywords}
LLM inference, quantization, GGUF, llama.cpp, ARM, x86, SIMD, KV-cache,

\section{Introduction}
% ===========================================================================

On-device LLM inference has converged on a de facto standard: GGUF K-quant
formats~\cite{gguf} implemented in llama.cpp~\cite{llamacpp}, running on
CPU-only endpoints from mobile phones to edge servers.
Practitioners choose among seven variants (Q2\_K through Q8\_0, spanning
2.6--8.5~bpw) guided by a single heuristic inherited from GPU literature:
\emph{higher bit-width yields higher throughput.}
This paper shows that heuristic is wrong on every CPU backend tested, explains
exactly why at the kernel instruction level, and derives a predictive model
for when and how much it fails.

\textbf{Finding 1: Non-monotonic throughput, CPU-general.}
On ARM (Pixel~6a, Cortex-X1 NEON) and x86 (Intel i5-1235U, AVX2), the
six-bit variant Q6\_K is the \emph{slowest}: 3.53~tok/s on ARM and 6.8~tok/s
on x86. The two-bit Q2\_K is the \emph{fastest}: 7.49~tok/s on ARM and
14.1~tok/s on x86---a $2.1\times$ intra-format spread in the wrong direction.
The root cause is SIMD dequantization overhead.
Q6\_K stores weights split across two arrays (\texttt{ql[128]+qh[64]} per
256-weight superblock) and requires ${\approx}26$ NEON operations to reconstruct
each block; Q2\_K uses a single AND-mask and ${\approx}10$ operations
(static analysis of \texttt{arm/quants.c}, commit \texttt{1a29907}; Table~\ref{tab:superblock}).
At the same memory bandwidth, dequantization cost dominates, inverting the
expected ordering on both ISAs.

\textbf{Finding 2: CPU/GPU divide.}
Apple M4 Metal reverses the ordering entirely: Q4\_K\_S leads at
19.9~tok/s, Q8\_0 is slowest at 6.4~tok/s.
On the same chip, M4 CPU-only inference outperforms Metal for Q8\_0
($12.6$ vs.\ $6.4$~tok/s, $1.97\times$) and Q6\_K ($9.3$ vs.\ $7.0$~tok/s),
because Metal's dedicated 4-bit dispatch paths eliminate the dequantization
bottleneck that defines CPU behavior.
The non-monotonic ordering is a CPU property, not a model property: it
disappears the moment execution moves off general-purpose SIMD.

\textbf{Finding 3: KV-cache cliff with L2-predictable thresholds.}
Q2\_K throughput collapses $-48\%$ on ARM as context grows from 256 to 2048
tokens (onset at ctx=512), and $-50\%$ on x86 (onset at ctx=1300--1400).
Both thresholds are predicted analytically: the per-layer combined K+V
working set is $2048\times\text{ctx}$ bytes~\cite{llama32,gqa}; when this
overflows L2 ($512$~KB on Cortex-X1, $1.25$~MB on i5-1235U), attention
computation becomes DRAM-latency-bound.
The formula $\text{ctx}_\text{cliff}=L2/1024$ matches observation within 5\%
on both platforms. Q3\_K\_M is immune ($<\pm11\%$) because its higher FFN
compute cost masks the proportional impact of the attention overhead---a
compute-masking effect, not a cache property.

\textbf{Why it matters.}
These findings directly contradict the guidance embedded in llama.cpp
documentation, Ollama model cards, and practitioner blogs, which uniformly
recommend Q4--Q8 variants for quality and speed.
On CPU, Q6\_K is strictly dominated: slower than Q4\_K\_M on throughput
\emph{and} lower accuracy than Q4\_K\_S on BoolQ (65\% vs.\ 74\%).
Q2\_K is the fastest variant at short context but collapses $-48\%$ by
ctx=2048 and fails near-randomly on HellaSwag MCQ tasks (19\%,
below the 25\% random baseline).
Correct deployment requires hardware-specific guidance, not bit-width heuristics.

\textbf{Contributions.}
We present the first mechanistic, cross-platform characterization of GGUF
K-quant inference on CPU, covering ARM NEON, x86 AVX2, and Apple Metal across
1{,}200+ measurements on Llama~3.2~3B and Qwen~2.5~1.5B~\cite{qwen25}.
We derive a kernel-level explanation of the throughput inversion (SIMD
superblock dequantization cost), an analytical KV-cache overflow model that
predicts cliff thresholds from L2 size alone, and a CPU/GPU divide analysis
showing the inversion is CPU-general and Metal-absent.

\S2 surveys related work; \S3 describes methodology; \S4--5 present
throughput and context-scaling results; \S6 covers quality benchmarks;
\S7 gives the mechanistic analysis; \S8 covers cross-platform validation;
\S9 gives deployment recommendations; \S10 concludes.

\subsection{Post-Training Quantization Methods}

The dominant PTQ methods for GPU inference---GPTQ~\cite{gptq}, AWQ~\cite{awq},
SmoothQuant~\cite{smoothquant}, LLM.int8()~\cite{dettmers_int8}, QuIP\#~\cite{quip},
and SpQR~\cite{spqr}---all optimize for CUDA/Tensor Core execution paths where
higher-bit representations generally improve throughput. These methods assume
that dequantization cost is negligible (GPU fuses it with GEMM) and that
weight layout is irrelevant to speed. Both assumptions break on CPU.
Dettmers and Zettlemoyer~\cite{dettmers_kbit} derived scaling laws showing 4-bit
is Pareto-optimal for inference \emph{quality}---consistent with our Q4\_K\_S finding---
but these laws were derived on GPU and do not model CPU SIMD overheads.

Calibration-based quantization methods such as OBC~\cite{obc} and
SparseGPT~\cite{sparsegpt} use second-order Hessian information to identify
important weights, forming the theoretical basis for GGUF's imatrix calibration.
Our evaluation confirms the literature finding~\cite{dettmers_kbit} that calibration
benefit diminishes sharply below 3~bpw: BoolQ accuracy \emph{degrades} $-4$ to $-8$~pp
for Q2\_K-imatrix and Q3\_K\_M-imatrix, while 4-bit variants are unaffected.

Li et al.~\cite{li_eval} found significant accuracy gaps even at 4-bit
precision across a range of tasks, and showed that GGUF K-quant accuracy is
non-monotonic with bit-width on several benchmarks.
Dutta et al.~\cite{dutta_accuracy} demonstrated that perplexity-equivalent compressed
models produce measurably divergent task behavior.
Both findings support our conclusion that WikiText-2 perplexity is an insufficient
proxy: Q5\_K\_M through Q8\_0 differ by only 0.035 PPL nats yet span 10+ percentage
points on BoolQ and TruthfulQA.

\subsection{CPU Inference Systems and SIMD Optimization}

The roofline model~\cite{williams_roofline} predicts that single-batch LLM decode is
memory-bandwidth-limited, since each token requires loading all weight matrices once.
Na et al.~\cite{na_cpu_perf} confirmed this memory-bandwidth-limited regime for
server CPUs, profiling the LLM decode bottleneck across cache levels.
Pope et al.~\cite{pope_efficiently_scaling} and DeepSpeed-Inference~\cite{deepspeed}
established the same regime on GPU.
Our work extends this picture: \emph{within} GGUF K-quants, dequantization instruction
overhead creates throughput inversions invisible to a pure bandwidth model.
At the same memory bandwidth, Q2\_K's 4-operand AND-mask dequantization is
fundamentally cheaper than Q6\_K's 8-operand split-bit shuffle---a distinction
the standard roofline model cannot capture.

Georganas et al.~\cite{georganas_simd} established the methodology of counting SIMD
instruction sequences and cache effects on throughput for DNN operations.
Shen et al.~\cite{shen_llm_cpu} demonstrated INT4 weight-only quantization on x86
AVX2/AVX512 Xeon CPUs achieving competitive throughput, providing a baseline for
our x86 Q2\_K finding of 14.1~tok/s at ctx=256.
He et al.~\cite{he_cpu_inference} extended this with distributed INT8/INT4 inference
and KV cache quantization on Intel Xeon.
Most directly relevant to our mechanistic analysis, Gope et al.~\cite{gope_arm_kernels}
(Arm Ltd.) characterize dequantization overhead in group-quantized LLM kernels for
ARM CPUs, reporting $3$--$3.2\times$ improvements in prompt processing and
${\approx}2\times$ in autoregressive decoding over llama.cpp baselines,
achieved through a combination of optimized SIMD kernels, SIMD-aware weight packing,
fast decompression-path optimizations, and broader MMLA vector utilization---
directly confirming that superblock layout and dequantization path are primary
performance determinants, independent of bit-width.

Zhang et al.~\cite{nomad_attention} showed that at long contexts (16k tokens),
attention memory access becomes the dominant CPU bottleneck and is amenable
to SIMD table-lookup acceleration. Our KV-cache cliff occurs at much shorter contexts
(ctx=512 ARM, ctx=1300--1400 x86) because the L2 cache is smaller than the L3 assumed
in server-scale work; the underlying mechanism is the same.

\subsection{Mobile and Edge LLM Inference}

\textbf{MELTing Point}~\cite{meltingpoint} (Laskaridis et al., MobiCom 2024) is the
closest related work, benchmarking LLM inference on consumer mobile devices across
ARM/iOS/Android hardware. They find that 3-bit quantization achieves 24.77\% \emph{lower}
throughput than 4-bit on mobile hardware, observing throughput non-monotonicity across
bitwidth tiers but attributing it only broadly to dequantization overhead and cache
alignment without a kernel-level mechanistic analysis, and without reporting
context-length sensitivity or establishing the CPU/GPU divide.
Our work provides (1) the SIMD kernel explanation for the ordering, (2) the
filled-context cliff analysis, (3) cross-ISA validation, and (4) the
Metal CPU/GPU comparison that shows the ordering inversion is a CPU property.

Xiao et al.~\cite{xiao_mobile} conducted a comprehensive measurement study of LLM
inference across mobile SoCs, observing throughput variation across quantization formats
without analyzing intra-format dequantization cost as the causal mechanism.
MLC-LLM~\cite{mlcllm} demonstrated competitive mobile GPU inference via ML compilation;
their results differ fundamentally from the CPU path because Metal/Vulkan kernel
fusion eliminates the dequantization bottleneck entirely.
Zhang and Huang~\cite{zhang_cpu_beats_gpu} provide an independent data point showing
CPU outperforming mobile GPU for 1B-parameter llama.cpp inference on iPhone~15 Pro,
consistent with our M4 finding that Metal is slower than CPU for high-bit variants.

Chen et al.~\cite{chen_heterollm} profiled Snapdragon 8 Gen 3 heterogeneous cores
(CPU+GPU+NPU) for LLM inference, showing that CPU/GPU assignment and memory
bandwidth shape per-token throughput---directly relevant to our big.LITTLE thread
scaling and Metal CPU/GPU divide findings.
Nguyen and Nguyen~\cite{nguyen_sbc} benchmarked 25 quantized LLMs on ARM single-board
computers (Raspberry Pi 4/5), providing the closest embedded-ARM comparison baseline.
Tummalapalli et al.~\cite{tummalapalli_edge} measured sustained-load inference on
Qwen 2.5 1.5B (the exact model we use for cross-model validation), reporting 44\%
throughput loss from thermal throttling on iPhone 16 Pro---consistent with our
thermal characterization findings ($\approx$40\% drop under sustained no-cooldown load on Pixel~6a).

Wang et al.~\cite{wang_biglittle} characterized CNN throughput on ARM big.LITTLE
clusters (Cortex-A57+A53), showing that cross-cluster scheduling overhead limits
scaling beyond the big-core count---establishing the prior work context for our
thread-scaling result: peak at 4 threads, regression at 8 (big-core saturation
followed by LITTLE-core contention).
Benoit-Cattin and Velasco-Montero~\cite{benoit_thermal} quantified CPU frequency
reduction from thermal throttling on Raspberry Pi 4B (Cortex-A72) under sustained
inference---providing the foundational empirical baseline for our Pixel~6a
thermal characterization.

MobileLLM~\cite{mobilellm} studies architecture choices for sub-billion parameter
on-device models. Our KV-cache formula is derived from Llama~3.2~3B's Grouped
Query Attention (GQA) configuration~\cite{llama32,gqa}; the formula generalizes
directly from the GQA architecture's number of KV heads.
EdgeMoE~\cite{edgeMoE} and PowerInfer~\cite{powerinfer} target MoE and CPU/GPU split
execution respectively; both require architectural modifications not applicable to
standard dense Llama/Qwen models.

\subsection{KV-Cache Memory and Context Length}

FlashAttention~\cite{flashattention,flashattention2} optimized attention tiling
for GPU SRAM reuse---a GPU-specific technique that does not transfer to CPU where
no on-chip SRAM is available at this scale.
vLLM~\cite{vllm} introduced PagedAttention for server KV-cache management;
our single-session edge use case does not have the fragmentation problem PagedAttention
solves, but the analysis of KV-cache working set size translates directly.
KVQuant~\cite{kvquant} proposed quantizing the KV cache itself; our work establishes
that L2 overflow at ctx=512 (ARM) and ctx=1300--1400 (x86) causes 40--50\% throughput
degradation even \emph{before} KV cache quantization, motivating KVQuant-style compression
specifically for CPU deployments.
Alizadeh et al.~\cite{llm_flash} highlighted memory access granularity as the key
bandwidth parameter for flash-storage LLM inference---analogous to our cache-line
analysis of the L2 cliff threshold formula.

\subsection{Quantization Evaluation and llama.cpp}

The standard benchmarks we use---BoolQ~\cite{boolq}, HellaSwag~\cite{hellaswag},
MMLU~\cite{mmlu}, TruthfulQA~\cite{truthfulqa}, ARC~\cite{arc},
WikiText-2~\cite{wikitext2}---are established NLP evaluation datasets.
The Zhou et al.\ survey~\cite{zhou_survey} provides the broader context for
efficient LLM inference methods, of which GGUF K-quant CPU inference is a
practically important but undercharacterized case.

The llama.cpp project~\cite{llamacpp} supports Android via the NDK, compiling
NEON SIMD kernels for arm64-v8a. The GGUF format~\cite{gguf} standardizes
K-quant weight layout with superblock structure.
Prior community llama.cpp benchmarks exist as informal GitHub issues and forum
posts, lacking controlled multi-trial methodology, context-length sweeps, and
mechanistic explanation. This work provides the first reproducible, scripted
evaluation with kernel-level analysis of the non-monotonic throughput ordering.

\section{Methodology}\label{sec:methodology}
% ===========================================================================

\subsection{Platforms}

\textbf{Primary platform (ARM):} A \textbf{Google Pixel~6a} (Android~16, API~36) with
the \textbf{Google Tensor G1} SoC: 2$\times$ Cortex-X1 big cores (512~KB L2 each),
2$\times$ Cortex-A76 mid, 4$\times$ Cortex-A55 little, 6~GB LPDDR5.
All main throughput, context-scaling, quality, and perplexity experiments are on this device.

\textbf{Cross-platform validation:}
\begin{itemize}[nosep]
  \item \textbf{x86 AVX2:} Intel i5-1235U laptop (12th Gen, 1.25~MB P-core L2,
    Windows~11, 6 threads, \texttt{llama-cli -ngl~0}).
    Device plugged into AC power throughout all measurements (performance mode, no
    battery throttling). No formal thermal characterization was performed on x86;
    a 60-second inter-variant cooldown was applied.
    Used for TPS, filled-context cliff sweep (all 7 variants $\times$ 11 ctx
    $\times$ 5 trials), and all 6 quality benchmarks ($n{=}100$ each).
  \item \textbf{Apple M4 Metal GPU:} MacBook Air M4 (16-core GPU, unified memory,
    \texttt{llama-bench -ngl~99}). Used for TPS ($n{=}10$) and full filled-context
    cliff sweep (all 7 variants $\times$ 13 ctx $\times$ 5 trials).
\end{itemize}

\textbf{Generalizability check:} \textbf{Qwen~2.5~1.5B~Instruct}~\cite{qwen25} (all 7 GGUF K-quant
variants) was benchmarked on the Pixel~6a ($n{=}20$ trials, ctx=256) to confirm the
non-monotonic ordering is model-independent.

\subsection{Model and Quantization Variants}

The model under evaluation is \textbf{Llama 3.2 3B Instruct}~\cite{llama32},
Meta AI's 3-billion-parameter instruction-tuned model. We evaluate seven quantization
variants available in the GGUF format~\cite{gguf}, spanning the full range from
2-bit to 16-bit precision:

\begin{itemize}
  \item \textbf{Q2\_K}: 2-bit K-quant, 1.3~GB on disk (3.40~bpw).
  \item \textbf{Q3\_K\_M}: 3-bit K-quant (medium), 1.6~GB.
  \item \textbf{Q4\_K\_S}: 4-bit K-quant (small), 1.56~GB (4.85~bpw). Evaluated for quality benchmarks.
  \item \textbf{Q4\_K\_M}: 4-bit K-quant (medium), 2.0~GB. Community standard.
  \item \textbf{Q6\_K}: 6-bit K-quant, 2.7~GB (6.59~bpw). Near-lossless compression.
  \item \textbf{Q8\_0}: 8-bit quantization, 3.4~GB. Heaviest quantized variant.
  \item \textbf{F16}: 16-bit floating point, 6.4~GB. Full reference precision.
\end{itemize}

\subsection{Experiment Design}

\textbf{Filled-context methodology.}
All throughput and context-scaling experiments use a \emph{filled-context} protocol:
for each target context size $N$, the input prompt is padded to exactly $N - 64$ tokens,
ensuring the KV cache is saturated before measurement begins.
This differs critically from the na\"ive approach of varying only the \texttt{-c} flag
while using a short fixed prompt, which measures allocation overhead rather than genuine
KV-cache pressure.
Results labelled ``ctx=N'' indicate a \emph{filled} N-token KV cache, not merely a window
size allocation.

\textbf{TPS sweeps.}
Decode throughput at ctx $\in$ \{256, 512, 1024, 2048\} with $n{=}10$ trials per
(variant, context) cell; trial~1 excluded as warmup; 5-minute cool-down between variants.
ARM Pixel~6a and x86: \texttt{llama-cli} with \texttt{-ngl~0} (CPU-only);
M4 Metal: \texttt{llama-bench} with \texttt{-ngl~99}.

\textbf{Cliff sweeps.}
Context sensitivity at 11 context sizes
(256, 384, 512, 640, 768, 896, 1024, 1280, 1536, 1792, 2048 tokens),
$n{=}10$ trials (ARM), $n{=}5$ (x86 and M4 Metal);
all 7 variants; filled-context methodology throughout.

\textbf{Cross-model replication.}
Qwen~2.5~1.5B~\cite{qwen25} repeated the ARM TPS sweep ($n{=}20$, ctx=256) and
the cliff sweep ($n{=}5$, 11 ctx) to confirm non-monotonic ordering is
model-independent.

\textbf{Thread scaling.}
Q4\_K\_M at ctx=256, 1/2/4/8 threads, $n{=}15$ trials, to characterize
big.LITTLE heterogeneous-core behavior on the Pixel~6a.

In total, the experiment produced \textbf{1,200+ valid measurements} across throughput,
context-scaling, and quality benchmarks, after discarding warmup runs and one contaminated
Q8\_0 ctx=2048 run (concurrent-process interference; replaced by a clean re-run).

\subsection{Metrics}

We measure the following metrics, extracted from llama-cli's structured timing output:
\begin{itemize}
  \item \textbf{Decode throughput} (tokens/s): Rate of output token generation.
        Primary metric for interactive user experience.
  \item \textbf{Prefill throughput} (tokens/s): Rate of input prompt processing.
  \item \textbf{Time-to-first-token (TTFT)} (s): Wall-clock time to first output
        token. Directly impacts perceived responsiveness.
  \item \textbf{End-to-end latency} (s): Total time for prefill + decode of 128
        output tokens.
  \item \textbf{BoolQ accuracy} (\%): Exact-match accuracy on a 100-question
        BoolQ reading comprehension benchmark (yes/no questions with passages).
  \item \textbf{ARC-Easy accuracy} (\%): 4-choice accuracy on a 100-question
        ARC-Easy elementary science benchmark.
\end{itemize}

\subsection{Quality Evaluation}

To characterize the accuracy ordering (§\ref{sec:quality}), we conducted two
standard benchmark evaluations directly on-device via USB ADB for all seven variants
(including F16).

\textbf{BoolQ (100 questions):} Reading comprehension benchmark~\cite{boolq}
drawn from the SuperGLUE validation split. Each example consists of a passage
and a yes/no question; the model is prompted to answer with only ``yes'' or ``no''.
Responses are scored via case-insensitive first-match extraction of the words
\emph{yes} or \emph{no} from the model output. The 100-question subset yields
a 95\% Wilson CI of $\pm$8--9\%, sufficient to distinguish variants that differ
by $>$7\%.

\textbf{ARC-Easy (100 questions):} Elementary science 4-choice benchmark~\cite{arc}
from the AI2 ARC Easy test split. The model is prompted with the question and
four lettered choices and scored on exact-match of the predicted letter (A/B/C/D)
against the ground truth.

\subsection{Repeatability Verification}

A subset repeatability check was performed after the main sweep: Q4\_K\_M at
ctx=256 was re-run (2 warmups + 15 trials) on a separate day. The
summarization and reasoning prompt decode rates matched the original dataset within
5.6\% ($5.47 \pm 0.12$ vs.\ $5.18 \pm 0.06$~tokens/s), which is within the
expected day-to-day variability of mobile hardware ($\pm$10--15\%) due to
OS scheduling and thermal starting conditions. One trial exhibited a transient
drop to 3.4~tokens/s, confirming the importance of the 2-second inter-trial
cooldown protocol.

\subsection{Framework Selection and Build}

We selected llama.cpp~\cite{llamacpp} as the inference backend after evaluating
ExecuTorch~\cite{executorch}: ExecuTorch's \texttt{LlmModule.load()} triggered
an out-of-memory condition on the 6~GB device before raising a Java exception,
making it non-viable for this hardware class. llama.cpp was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a with NEON SIMD kernels.
The llama-cli binary was pushed to the device via \texttt{adb push} and
invoked through WiFi ADB to avoid USB charging interference with thermal
measurements. The default thread count is 4 (matching the number of big+mid
cores: 2$\times$ X1 + 2$\times$ A76) unless otherwise stated.

\section{Context Sensitivity: The KV-Cache Cliff}
\label{sec:cliff}

\begin{figure*}[t]
  \centering
  \includegraphics[width=\textwidth]{../figures/fig_cliff_3plat_sel.png}
  \caption{Decode throughput (tok/s) vs.\ context window under \emph{filled-context}
  conditions (prompt $\approx N{-}64$ tokens; error bars = SEM; 4 representative variants).
  \textbf{(a) ARM Pixel~6a} (512~KB L2 per core): Q2\_K collapses $-48\%$ (cliff at ctx=512,
  matching $L2/1024$ exactly); Q3\_K\_M stable ($<\pm11\%$); red shading = ARM L2 cliff zone.
  \textbf{(b) x86 i5-1235U} (1.25~MB L2): Q2\_K collapses $-50\%$, but cliff is
  delayed to ctx$\approx$1300--1400 by the larger L2; Q3\_K\_M and Q4\_K\_M flat ($<\pm8\%$).
  \textbf{(c) M4 Metal} (GPU unified memory): all variants flat within $\pm2\%$---no cliff.
  The cliff threshold tracks L2 size ($\approx$L2/2048 tokens); Metal eliminates it entirely.}
  \label{fig:kv_cliff}
\end{figure*}

\textbf{Methodology note.} Prior work often sweeps context by varying the KV
allocation parameter~\texttt{-c N} while using a short fixed prompt.  This
measures only allocation overhead, not genuine KV-cache pressure.  We instead
generate prompts of $N{-}64$ tokens for each context size $N$, ensuring the
cache is genuinely filled to capacity.  This \emph{filled-context} methodology
produces qualitatively different results from the allocation-only baseline.

Context-length sensitivity is \textbf{non-monotonic with quantization level}---
mirroring the throughput ordering anomaly from §\ref{sec:throughput}.
Table~\ref{tab:ctx2048} summarises decode throughput from ctx=256 to ctx=2048.

The striking finding is that \textbf{Q2\_K, the fastest variant at short context,
is also the most fragile at long context}: decode throughput collapses from
7.07~tok/s (ctx=256) to 3.69~tok/s (ctx=2048), a $-48\%$ degradation, with
the cliff onset at ctx=512 ($-27\%$)---exactly matching the formula prediction of
$512\text{~KB}/1024 = 512$~tokens.
By contrast, \textbf{Q3\_K\_M is the most context-stable variant}: throughput
holds near $4.07$~tok/s from ctx=256 to ctx=1024, declining to $3.62$~tok/s
at ctx=2048 ($-11\%$ over an $8{\times}$ context increase).

Mid-precision variants split into two groups.
\textbf{Q5\_K\_M shares Q2\_K's early cliff}: throughput drops $-46\%$
(6.67 to 3.61~tok/s), with onset at ctx=512 ($-18\%$ single step)---matching
the ARM L2 threshold.
An earlier n=3 estimate ($-26\%$, cliff at ctx=1300) was an artifact of a
cold-device baseline that fell near the cliff floor; the canonical n=5 isolated run
(2026-04-10) and n=10 rerun data are consistent at $-18$--$20\%$ at ctx=512.
Note: the cliff-sweep baseline (6.67~tok/s) exceeds the fresh-context TPS value
(3.75~tok/s, Table~\ref{tab:tps_results}); the difference reflects DVFS behavior---
the 192-token filled prefill sustains CPU turbo frequency before decode starts,
whereas the TPS sweep's 7-token prompt does not.
Q8\_0 loses $-18\%$ (4.53 to 3.71~tok/s, cliff at ctx=1300) and Q4\_K\_S $-10\%$
(cliff at ctx=1200).
Q6\_K shows mild degradation of $-11\%$ (3.55 to 3.17~tok/s),
and Q4\_K\_M loses only $-6\%$ (5.57 to 5.21~tok/s)---both essentially stable.

This pattern is mechanistically informative.
The Llama~3.2~3B KV cache at 2048 tokens occupies $\approx$224~MB
(2 $\times$ 28 layers $\times$ 8 heads $\times$ 128 head-dim $\times$ 2048 $\times$ 2 bytes).
For Q2\_K, whose weight file is only 1.28~GB, this KV footprint represents
$\approx$17.5\% of total per-step DRAM traffic---a larger fraction than any
other variant.  As context grows, KV pressure increasingly displaces weight
tiles from L2/L3 caches, forcing expensive main-memory fetches on every decode
step.  Q3\_K\_M (1.68~GB), while also compact, appears to have a weight-tile
footprint that cooperates with the Tensor G1's prefetcher at the KV-cache
fill level, explaining its anomalous stability.

\textbf{Recommendation}: for deployments requiring ctx$\geq$512, prefer
Q3\_K\_M (${\approx}4.0$~tok/s, $-11\%$ to ctx=2048) or Q4\_K\_M (${\approx}5.2$~tok/s,
$-7\%$) over Q2\_K, whose $-48\%$ cliff begins at ctx=512.
Q5\_K\_M behaves like Q2\_K (cliff at ctx=512, $-46\%$ total): avoid for ctx$\geq$512
unless paired with KV Q8\_0 quantization.
Q8\_0 offers good mid-context stability ($-18\%$ to ctx=2048).

Note: Flash Attention (\texttt{-fa} flag) was tested as a potential mitigation
for the Q2\_K cliff but was found to be unsupported by the \texttt{llama-completion}
binary build used in this study.  Evaluating FA-enabled builds is left for future work.

\label{sec:quality}

  Q8\_0 (68\%, 4.52~tok/s) is dominated by Q4\_K\_M on both axes.}
  \label{fig:pareto}
\end{figure}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig9_model_size_vs_decode_tps.png}
  \caption{Model size (GB) vs. decode throughput (tokens/s). Q4\_K\_M achieves

  \label{fig:size_vs_tps}
\end{figure}

BoolQ reading comprehension accuracy was measured for all seven variants via USB
ADB (100 questions per variant; 95\% Wilson CIs are shown in Table~\ref{tab:main_results}).
Results: Q4\_K\_S 74\% (CI: 64--82\%), Q4\_K\_M 72\% (63--80\%), Q2\_K 69\% (59--77\%),
Q3\_K\_M 69\% (59--77\%), Q8\_0 68\% (58--76\%), Q5\_K\_M 67\% (57--76\%),
Q6\_K 65\% (55--74\%).

\textbf{Statistical separability.}
With $n{=}100$, the Wilson 95\% CI is $\pm$8--9pp per variant.
A two-proportion $z$-test (pooled SE) yields the following:
the Q4\_K\_S (74\%) vs.\ Q6\_K (65\%) gap ($z{=}1.38$, $p{\approx}0.17$) is
directionally consistent but not individually significant at $\alpha{=}0.05$.
No pairwise BoolQ comparison reaches significance at $n{=}100$.
Confident individual claims are therefore inappropriate; we instead report
\emph{aggregate patterns}:
\begin{itemize}[nosep]
  \item The \textbf{Q6\_K Pareto dominance failure} is robust: Q6\_K is simultaneously the
        lowest-scoring (65\%) and slowest (3.53~tok/s) practical variant---dominated on
        both axes---regardless of CI overlap.
  \item The \textbf{4-bit cluster} (Q4\_K\_S, Q4\_K\_M: 72--74\%) is directionally above
        the 6-bit cluster (Q6\_K: 65\%) with consistent margins across both BoolQ and ARC-Easy.
  \item Q8\_0 offers no accuracy advantage over Q4\_K\_M despite 70\% more weight bits.
\end{itemize}
ARC-Easy accuracy ranges 76--82\% across all seven variants
(Q4\_K\_M highest at 82\%, Q2\_K lowest at 76\%, all verified with the
current extracted-letter evaluation script).
The 6-point spread is narrower than BoolQ's 9-point spread, confirming
that BoolQ---which requires cross-sentence reasoning---is the more
informative quality discriminator among variants at this model scale.

\textbf{Instruction-following collapse at extreme quantization.}
HellaSwag---a four-choice sentence completion task requiring selection of a lettered
option (A/B/C/D)---reveals a qualitatively different and mechanistically distinct
failure mode at Q2\_K.
Per-question output analysis of all 100 HellaSwag responses shows:
Q2\_K outputs ``No'' for 56 of 100 questions (56\%), with only 31 valid ``A''
responses and 9 distributed across B/C/D.
The expected answer for these 56 ``No'' outputs is uniformly distributed across
A, B, C, D (21/11/11/13), confirming this is not a label-bias artifact but a
\emph{prompt-format confusion}: at Q2\_K's 3.40 bits per weight, the model
interprets the HellaSwag answer slot as a binary yes/no question rather than
a lettered-choice selector.
The result is 19/100 correct (19\%), below the 25\% random baseline (the 31 ``A''
responses are partly correct by chance), compared to 39--45\% for all
higher-precision variants where letter-selection works correctly.
In contrast, Q4\_K\_M outputs 20 non-letter responses but mostly valid ``A''
choices (58/100), achieving 43\%.

This collapse does not appear on BoolQ (yes/no format, Q2\_K scores 69\%)
or ARC-Easy (76\%), confirming the failure is \emph{prompt-format-specific}:
at this compression level, the model retains factual recall and binary judgment
but conflates non-binary multi-choice prompts with binary question formats.
Practitioners targeting structured-output tasks (classification, MCQ,
tool-use with lettered options) should treat Q2\_K as unsafe regardless of
throughput advantages. The $z$-test for Q2\_K (19\%) vs.\ the next-lowest
Q4\_K\_S (39\%) gives $z{=}3.13$, $p{<}0.002$---the only statistically
significant pairwise quality result in this study.

Figure~\ref{fig:pareto} plots the Pareto frontier for all variants.
Q4\_K\_S (74\%, 5.01~tok/s) achieves the highest BoolQ accuracy of any variant,
while Q2\_K (69\%, 7.49~tok/s) leads on raw decode throughput.
Q4\_K\_M (72\%, 4.78~tok/s) provides a stable middle ground with the lowest
context-length sensitivity.
Q6\_K is strictly dominated on every metric: slower than Q4\_K\_M,
less accurate than Q4\_K\_S, and no more context-stable.
Notably, \textbf{Q8\_0 is no longer Pareto-optimal under BoolQ}: it scores 68\%
accuracy at 4.52~tokens/s, dominated by Q4\_K\_M which achieves both higher
accuracy \emph{and} higher throughput. Q3\_K\_M (69\%, 4.68~tokens/s) is also
dominated by Q4\_K\_M on BoolQ vs.\ throughput, though Q3\_K\_M leads on
context stability and TruthfulQA (see below). \textbf{Q6\_K (65\%, 3.53~tokens/s) is
strictly dominated on both dimensions}---simultaneously the lowest-scoring and
slowest practical variant.

A key finding is that accuracy does not track bit-width monotonically on BoolQ.
Q4\_K\_S (4-bit, small) achieves the \emph{highest} accuracy at 74\%;
Q4\_K\_M (4-bit, medium) follows at 72\%; Q6\_K (6-bit)
scores \emph{lowest} at 65\%---fewer correct answers than any other variant despite
having more bits per weight than either 4-bit variant.
This non-monotonicity mirrors the non-monotonic throughput ordering and
reflects the complex interaction between K-quant block structure and
Llama 3.2's layer-specific weight sensitivity.

\textbf{MMLU accuracy.}
A 100-question MMLU evaluation was conducted across all quantized variants.
Results: Q2\_K=42\%, Q3\_K\_M=48\%, Q4\_K\_S=49\%, Q4\_K\_M=47\%, Q5\_K\_M=50\%, Q6\_K=48\%, Q8\_0=47\%.
MMLU accuracy clusters tightly between 42--50\% across all variants
(95\% Wilson CI width $\approx\pm10$pp), making pairwise distinctions statistically
unreliable at $n{=}100$ questions. Q2\_K is detectably weaker (42\%), while
Q3\_K\_M through Q8\_0 are statistically indistinguishable on MMLU alone.
Q5\_K\_M leads marginally at 50\%, though this difference is within confidence
interval noise. These results confirm that MMLU at $n{=}100$ is insufficient to
rank quantization variants and that BoolQ is the more discriminative on-device
quality benchmark for this model.

\textbf{TruthfulQA accuracy.}
A 100-question TruthfulQA multiple-choice (MC1) evaluation was conducted across all
seven variants, measuring factual consistency and resistance to common misconceptions.
Results: Q3\_K\_M=68\%, Q5\_K\_M=65\%, Q4\_K\_M=60\%, Q6\_K=60\%, Q8\_0=58\%,
Q4\_K\_S=57\%, Q2\_K=50\%.
Three findings are noteworthy.
First, \textbf{Q3\_K\_M leads at 68\%}---the highest truthfulness score despite
being mid-range on BoolQ (69\%) and slower than Q2\_K. This is the same variant
that shows near-zero context sensitivity (§\ref{sec:cliff}), suggesting a
consistent pattern: Q3\_K\_M's block quantization structure may preserve factual
representations better than its bit-count implies.
Second, \textbf{Q2\_K collapses to 50\%}---near-chance for binary MC1---while
scoring 69\% on BoolQ and exhibiting the HellaSwag instruction-following collapse (19\%).
These three failure modes (HellaSwag format, TruthfulQA factual coherence, long-context
throughput) converge on Q2\_K as a variant with fundamental limitations beyond its
throughput advantage.
Third, the \textbf{non-monotonic quality ordering extends to TruthfulQA}: Q3\_K\_M (3.95~bpw) outperforms
Q4\_K\_S (4.85~bpw), Q8\_0 (8.5~bpw), and Q6\_K (6.59~bpw) on truthfulness.
This corroborates the BoolQ and MMLU findings that bits-per-weight is a poor predictor
of downstream task quality.

\subsection{Thread Scaling and Memory Footprint}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig4_peak_memory_vs_quant.png}
  \caption{Peak working memory footprint by quantization variant, measured as the
  MemAvailable delta in \texttt{/proc/meminfo} before and after inference. Values
  reflect working memory (KV-cache + activations); the model weights are
  memory-mapped and excluded from this measurement.}
  \label{fig:memory}
\end{figure}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig8_latency_distribution.png}
  \caption{Distribution of end-to-end latency across all trials for each quantization
  variant. Q3\_K\_M and Q6\_K exhibit notably tighter distributions than Q2\_K and Q8\_0.}
  \label{fig:latency_dist}
\end{figure}

\textbf{Thread-count scaling.} We measured Q4\_K\_M decode throughput at 1, 2, 4,
and 8 threads (256-token context, filled-context methodology, $n$=15 trials per count). Results are:
1 thread: 3.38~tokens/s;
2 threads: 3.78~tokens/s (1.12$\times$);
4 threads: 4.81~tokens/s (peak, 1.42$\times$);
8 threads: 3.83~tokens/s (1.13$\times$).

The Tensor G1's Cortex-X1 P-cores drive the single-thread baseline; two P-cores yield
only modest 12\% gains (memory-bandwidth-bound, not compute-bound). At 4 threads,
P-cores plus two A76 E-cores push throughput to the 4.81~tokens/s peak.
At 8 threads, the 4 Cortex-A55 little cores are recruited, but their lower
clock speed and memory bandwidth share introduces load imbalance---resulting in
a 20\% throughput \textit{regression} from 4 to 8 threads. This result
confirms that on heterogeneous ARM clusters, practitioners should not blindly
use all available cores; instead, thread count should match the number of
performance cores.

\textbf{Deployment feasibility by variant.}
Q2\_K (1.3~GB) and Q3\_K\_M (1.6~GB) are the most RAM-friendly and would
run on devices with as little as 3~GB total RAM.
Q2\_K is the pure throughput choice (7.49~tokens/s), but exhibits
instruction-following collapse on MCQ tasks (19\% HellaSwag),
collapses to near-chance on TruthfulQA (50\%), and degrades $-48\%$
at long context (cliff at ctx=512).
Q3\_K\_M (4.68~tokens/s, 69\% BoolQ, 68\% TruthfulQA) matches Q2\_K on
BoolQ, leads on factual consistency, and is context-stable ($-11\%$ ctx=256$\to$2048).
\textbf{Q3\_K\_M is the recommended choice at the 3~GB memory tier}
for any workload beyond short-context throughput-only use cases.
Q4\_K\_S (1.56~GB) and Q4\_K\_M (2.0~GB) comfortably fit any device $\geq$ 4~GB.
Q4\_K\_S achieves the highest BoolQ accuracy (74\%) among all variants; Q4\_K\_M
(72\%, 4.78~tokens/s) offers the most context-stable performance profile.

F16 (6.4~GB) is not viable on any consumer device $\leq$ 8~GB.

\section{Root Cause: SIMD Dequantization and Cache Pressure}
\label{sec:mechanism}

\subsection{SIMD Dequantization Overhead}
\label{sec:nonmonotonic}

\textbf{Superblock structure and kernel complexity.}
All K-quant variants share a 256-element superblock (\texttt{QK\_K=256}) in
\texttt{ggml-common.h} (llama.cpp commit \texttt{1a29907}).
Each superblock stores the weights in a \emph{split-bit} layout whose complexity
increases with bit-width, as shown in Table~\ref{tab:superblock}.
Q2\_K uses simple 2-bit masking (single \texttt{AND}+\texttt{SHR} per group).
Q6\_K stores weights across \emph{two separate arrays}: lower 4 bits in \texttt{ql[128]}
and upper 2 bits in \texttt{qh[64]}, requiring a 4-operation reconstruction
(\texttt{AND}, \texttt{SHR}, \texttt{SHL}, \texttt{OR}) per group---nearly
3$\times$ more bit-manipulation work per element than Q2\_K.

\textbf{Why Q6\_K is slower than Q2\_K and Q8\_0 on CPU.}
Inspecting \texttt{ggml\_vec\_dot\_q2\_K\_q8\_K} and \texttt{ggml\_vec\_dot\_q6\_K\_q8\_K}
in both \texttt{arm/quants.c} (NEON) and \texttt{x86/quants.c} (AVX2) reveals the same pattern.
The Q2\_K inner loop (unrolled for QK\_K/128 = 2 iterations) loads
\textbf{32~bytes} of weight data per 128 weights---one 256-bit AVX2 register---and
extracts four 2-bit groups via four \texttt{AND}/\texttt{SHR} pairs before the
\texttt{MADDUBS}/\texttt{MADD} dot products.
Q6\_K loads \textbf{96~bytes} per 128 weights (two 32-byte \texttt{ql} chunks plus
one 32-byte \texttt{qh} chunk), then executes eight bit-manipulation instructions
to reconstruct each 6-bit value before the same dot products.
The memory bandwidth difference (3$\times$ more weight bytes per compute iteration)
compounds with the extra reconstruction instructions, yielding a total AVX2 cost of
$\approx$70 instructions per 256-weight block vs.\ $\approx$40 for Q2\_K.
This explains why Q6\_K---despite a smaller model file than Q8\_0---is \emph{slower}:
its split-field reconstruction overhead exceeds the bandwidth savings from compression.

The AVX2 kernel confirms the same pattern seen on NEON, validating the
\textbf{CPU-general} (not ARM-specific) nature of the non-monotonic ordering.
Both ISAs share the same algorithmic cost structure: simple 2-bit masking (Q2\_K)
vs.\ complex multi-field reconstruction (Q6\_K).
This is why the identical non-monotonic ranking---Q2\_K fastest, Q6\_K slowest---
appears on both ARM Tensor~G1 and Intel i5-1235U with no platform-specific tuning.

\subsection{KV-Cache L2 Overflow and Cliff Threshold Prediction}
\label{sec:kvcache_mechanism}

\textbf{KV-cache working set per attention layer.}
Llama 3.2 3B uses Grouped Query Attention with $N_{kv}=8$ KV heads,
head dimension $D_{head}=128$ (from hidden size 3072 / 24 query heads), and $L=28$ layers.
The KV cache is stored in fp16 regardless of weight quantization variant.
The combined K+V footprint per layer grows linearly with context:
\begin{equation}
C_{\text{layer}}(\text{ctx}) = 2 \times N_{kv} \times D_{head} \times \text{ctx} \times 2~\text{bytes} = 4096 \times \text{ctx}~\text{bytes}
\label{eq:kvcache}
\end{equation}
(factor of 2: K+V; factor of 2 bytes: fp16 precision).
During decode, the attention kernel streams this per-layer buffer for each token generated.

\textbf{Empirically validated cliff threshold formula.}
Although the full per-layer K+V footprint is $4096\times\text{ctx}$ bytes,
the observed cliff threshold is well described by the simpler empirical formula:
\begin{equation}
\text{ctx}_{\text{cliff}} \approx \frac{L2_{\text{cache}}}{1024}
\label{eq:cliff}
\end{equation}
The 1024 denominator is \emph{empirically calibrated}: it is consistent with the
attention kernel's tiled memory access pattern, which keeps only a portion of the KV
buffer resident in L2 at any time (due to interleaved access across the 8 KV groups
and L2 sharing with instruction cache and activation buffers).

Validation on two platforms:
\begin{itemize}[nosep]
  \item \textbf{ARM Cortex-X1} (512~KB L2 per core):
        $512{,}288 / 1024 = 512~\text{tokens}$---observed cliff onset ctx=512 (n=10). \textbf{0\% error.}
  \item \textbf{Intel i5-1235U} (1.25~MB P-core L2):
        $1{,}310{,}720 / 1024 = 1280~\text{tokens}$---observed cliff onset ctx=1300--1400 (n=5). \textbf{Within 8\%.}
\end{itemize}
When ctx exceeds $\text{ctx}_{\text{cliff}}$, the attention \texttt{SDOT}/\texttt{MADDUBS}
operations stall on DRAM latency instead of L2 hits, causing Q2\_K throughput to collapse
$-48\%$ on ARM and $-50\%$ on x86.

\textbf{Why Q3\_K\_M shows no cliff ($<\pm11\%$ across all contexts).}
The key insight is that the KV-cache attention time is the \emph{same} for all variants
(same model architecture, same fp16 KV cache, same context length).
However, the total decode time per token is:
\[T_{\text{decode}} = T_{\text{attn}}(\text{ctx}) + T_{\text{ffn}}\]
where $T_{\text{ffn}}$ is the feed-forward network cost (dominated by weight dequantization).
Q2\_K has the cheapest dequantization ($\approx$10 NEON ops/256w), making $T_{\text{ffn}}$
small, so $T_{\text{attn}}$ constitutes a large fraction of $T_{\text{decode}}$.
When L2 overflow doubles $T_{\text{attn}}$, the percentage drop in throughput is large.
Q3\_K\_M's hmask-based dequantization ($\approx$18 NEON ops/256w, 31\% more instructions
and 1.3$\times$ more bytes loaded than Q2\_K) increases $T_{\text{ffn}}$ proportionally.
The same absolute increase in $T_{\text{attn}}$ from KV cache overflow
becomes a smaller fraction of the larger $T_{\text{ffn}}+T_{\text{attn}}$ total,
attenuating the cliff to $<\pm11\%$ across all contexts (observed: $-10.9\%$ ARM,
$-11.2\%$ x86).
This is not intrinsic immunity; it is a masking effect from Q3\_K\_M's higher
baseline compute cost per token.

Using 8 threads degrades throughput $-20\%$ due to little-core load imbalance

\textbf{Q6\_K.}
Strictly dominated on every metric: slower than Q4\_K\_M (3.53 vs.\ 4.78~tok/s),

KV Q8\_0 (\texttt{-ctk q8\_0 -ctv q8\_0}) eliminates the Q2\_K cliff entirely

for ctx$\geq$512; Q8\_0 ($-18\%$, cliff ctx=1300) is the safer mid-precision
choice for extended context.
Q6\_K shows mild degradation ($-11\%$) and Q4\_K\_S moderate ($-15\%$).

\textbf{Use KV-cache Q8\_0 quantization for long-context deployments.}
Enabling \texttt{-ctk q8\_0 -ctv q8\_0} eliminates the Q2\_K cliff entirely
(see \S\ref{sec:kvcache_quant}) but reduces peak short-context throughput by $-46\%$.
For Q3\_K\_M and Q4\_K\_M, KV Q8\_0 is a net win above ctx=512 with $<$1\% short-context
penalty. The recommended configuration for ctx$\geq$512 is \textbf{Q4\_K\_M + KV Q8\_0}:
4.70~tok/s at ctx=256 and 4.44~tok/s at ctx=2048 ($-5.5\%$),
outperforming default Q4\_K\_M at every context above 1024 tokens.

\textbf{TTFT note.}
TTFT grows with KV-cache window size across all variants (larger cache initialization cost).
Responsiveness-sensitive deployments should budget TTFT alongside decode throughput.

Two findings stand out.
\textbf{First}, the Q2\_K HellaSwag collapse \emph{replicates exactly on x86}:
19\% (ARM) and 22\% (x86)---both near or below the 25\% 4-choice random baseline---
confirming this is a \textbf{model-level failure mode}, not a runtime or platform artifact.
\textbf{Second}, MMLU accuracy is near-perfectly consistent across platforms
(ARM vs.\ x86 differs by $\leq$3\% for every variant), validating that quality
measurement methodology is reproducible across ISAs.
BoolQ and TruthfulQA patterns replicate as well: Q3\_K\_M leads on TruthfulQA on both
platforms (ARM 68\%, x86 60\%), and Q2\_K remains at near-chance (ARM 50\%, x86 49\%).
HellaSwag scores are slightly higher on x86 for Q4+~variants (39--45\%~ARM vs.\
47--55\%~x86); given overlapping Wilson 95\% CI bands, we attribute this to
between-sample variance rather than a genuine platform effect.
Taken together, the x86 quality results confirm that quantization-induced failure
modes are \textbf{runtime-independent} and can be characterized once and applied
across CPU architectures.

\textbf{M4 CPU is a third data point: non-monotonic, but different from ARM.}
Running llama-bench with \texttt{-ngl~0} (CPU-only, no Metal offload) on the same M4 Mac
reveals a third non-monotonic ordering: Q4\_K\_S leads at 13.2~tok/s, followed by
Q8\_0 (12.6~tok/s) and Q4\_K\_M (12.5~tok/s); Q6\_K is again slowest at 9.3~tok/s.
M4 CPU is 1.4--2.6$\times$ faster than Pixel~6a CPU (same 4-thread config), reflecting
the M4's higher IPC and memory bandwidth.
Q6\_K is slowest on \emph{all three} CPU backends (ARM, x86, M4)---confirming the
SIMD dequantization overhead mechanism is not architecture-specific.

\textbf{M4 CPU shows no structured KV-cache cliff.}
A dedicated filled-context cliff sweep (13~ctx points $\times$ 5~trials, script
\texttt{m4\_cpu\_cliff.sh}, \texttt{ngl=0}, 4~threads) confirmed no ARM-style cliff
in the ctx=256--2048 range.
This is consistent with the $L2/1024$ formula: the Apple M4's large per-cluster
L2 (${\approx}16$~MB) predicts a cliff at ctx${}\approx{}16{,}384$~tokens---well
beyond our test range.
However, the M4 CPU sweep exhibited high context-to-context variance
(coefficient of variation up to 80\% for some variants and context points) attributed
to macOS background scheduling and AMX dispatch path changes at certain working-set
sizes.
Reliable baseline TPS values are drawn from the dedicated TPS sweep ($n{=}10$,
Table~\ref{tab:crossdevice}); the cliff sweep corroborates the ordering but is
not used for quantitative throughput claims.

\textbf{Metal reverses the ordering---and is slower than M4 CPU for Q8\_0 and Q6\_K.}
Q4\_K\_S leads at 19.9~tok/s and Q4\_K\_M at 19.2~tok/s on Metal;
Q8\_0 (6.4~tok/s) and Q6\_K (7.0~tok/s) are slowest---opposite of M4 CPU.
Crucially, Metal is \textbf{slower than M4 CPU} for both variants:
Q8\_0 GPU/CPU = $6.39/12.60 = 0.51\times$ and Q6\_K GPU/CPU = $7.02/9.29 = 0.76\times$.
Metal has dedicated 4-bit kernel paths that accelerate Q4\_K\_S ($1.51\times$ GPU vs.\ CPU);
Q8\_0 requires only trivial byte-unpack on CPU, making GPU dispatch overhead \emph{net negative}.
Q2\_K, dominant on CPU backends, falls to rank~3 on Metal (17.8~tok/s, $1.45\times$ GPU/CPU).
The throughput ratio ARM$\to$Metal (Q4\_K\_M: $19.22 / 4.78 = 4.0\times$)
matches the T4 GPU reference baseline in Table~\ref{tab:gpu_baseline},
confirming Metal performance is quantitatively comparable to a shared T4.

\textbf{Metal eliminates the KV-cache cliff entirely.}
Figure~\ref{fig:kv_cliff} shows the contrast directly.
On ARM, Q2\_K degrades $-48\%$ and other variants $-11$--$-26\%$ across
ctx=256--2048; on x86, Q2\_K degrades $-50\%$.
On M4 Metal (right panel), all seven variants are flat or improving across
ctx=1024--2048 under the same filled-context methodology ($n{=}5$ trials,
complete cliff sweep in \texttt{m4\_metal\_cliff\_20260323\_015934}).
Five variants (Q4\_K\_S, Q4\_K\_M, Q5\_K\_M, Q6\_K, Q8\_0) vary by only $\pm0.6$--$0.8\%$---
below measurement noise.
Q3\_K\_M drifts $+3.4\%$ and Q2\_K increases $+8.5\%$ (9.40$\to$10.20~tok/s);
both are monotone \emph{improvements}, not degradation.
Metal performance is insensitive to context length because GPU unified memory
operates at much higher bandwidth ($>$400~GB/s on M4) than any CPU's L2 or L3
cache, so the KV-cache working set never becomes a bottleneck regardless of context size.
This is the clearest possible manifestation of the CPU/GPU divide:
the same cache-overflow mechanism that degrades CPU inference by 48--50\%
has zero effect on Metal inference.

\textbf{Unified interpretation.}
Four platforms establish a clean \textbf{CPU/GPU divide}:
\emph{all CPU backends} (ARM NEON, x86 AVX2, M4 CPU) share non-monotonic ordering
driven by SIMD dequantization overhead, with Q6\_K slowest on every CPU.
Only Metal GPU (with dedicated quantized-format dispatch paths) reverses toward 4-bit dominance.
The GPU/CPU comparison further sharpens the picture: Metal provides meaningful speedup
only for 4-bit variants ($1.5\times$); for Q8\_0 it is actively harmful ($0.51\times$).
Practitioners targeting CPU inference---whether ARM mobile, x86 desktop, or Apple Silicon CPU-only---should
expect Q6\_K to underperform; those targeting Metal (iOS, macOS) should use Q4\_K\_S or Q4\_K\_M
and \emph{avoid} Q8\_0, which loses $1.97\times$ to its CPU-only equivalent on the same hardware.

\textbf{Generalizability: Qwen 2.5 1.5B replicates the non-monotonic ordering.}
To test whether the non-monotonic CPU throughput ordering is Llama-specific, we ran
a TPS sweep ($n{=}5$ trials, ctx=256, filled-context, 4 threads) of all seven
K-quant variants of \textbf{Qwen~2.5~1.5B~Instruct}~\cite{qwen25} on the Pixel~6a
(run \texttt{pixel\_qwen\_tps\_20260326\_033619}).
The result is an \emph{exact replication} of the Llama~3.2~3B ordering:
\textbf{Q2\_K is fastest at 16.1~tok/s and Q6\_K is slowest at 7.20~tok/s},
a ratio of $2.23\times$---wider than Llama's $2.12\times$ spread (7.49/3.53).
The intermediate ordering (Q4\_K\_S=11.1, Q8\_0=10.2, Q3\_K\_M=10.1, Q4\_K\_M=9.78,
Q5\_K\_M=7.75~tok/s) is also non-monotonic in bit-width, confirming the pattern is
not an artefact of Llama's particular weight distribution.
The mechanism is the same in both models: all GGUF K-quant variants share the same
ARM NEON dequantization kernels; Q2\_K's compact block structure dequantizes faster
than Q6\_K's finer-grained 6-bit tiles regardless of model size, because the
bottleneck is the \emph{dequantization loop itself}, not the model weight dimensions.

\textit{KV-cache formula and cliff prediction for Qwen.}
Qwen~2.5~1.5B uses Grouped Query Attention with 28 layers, 2~KV heads, and
head dimension 128 (fp16 KV cache), giving a per-layer cache footprint of
$C_{\text{layer}}(\text{ctx}) = 2 \times 2 \times 128 \times \text{ctx} \times 2$~bytes
$= 1024 \times \text{ctx}$~bytes---\emph{half} that of Llama~3.2~3B ($2048 \times \text{ctx}$).
The Cortex-X1's 512~KB L2 therefore predicts a cliff at
ctx $\approx 524{,}288 / 1024 \approx 512$~tokens.

\textit{Cliff sweep: Q2\_K cliff confirmed at ctx=512.}
A dedicated sequential filled-context cliff sweep
(run \texttt{pixel\_qwen\_cliff\_filled\_20260330\_235410}, $n{=}5$/cell, 11~ctx points)
confirms the $L2/1024$ prediction precisely.
Q2\_K drops $-24.7\%$ from ctx=256 to ctx=512 (12.63$\to$9.51~tok/s)---the
largest single-step degradation---and reaches $-38.5\%$ total at ctx=2048.
All other variants are stable: Q3\_K\_M, Q4\_K\_S, Q4\_K\_M, Q5\_K\_M, and Q6\_K
each degrade $\leq10\%$ total with no sharp step.
Q8\_0 degrades $-16.6\%$---modest and monotone.
The Qwen result confirms that the cliff mechanism generalises beyond Llama~3.2~3B:
the formula $\text{ctx}_{\text{cliff}}\approx L2/1024$ correctly predicts ctx=512 for
a \emph{different} model (Qwen~2.5~1.5B) with a \emph{different} KV head count (2~vs.~8).

\section{Limitations and Future Work}
\label{sec:limitations}
\label{sec:future}
% ===========================================================================

\subsection{Current Limitations}

\textbf{Limited device diversity.} The ARM results are from a single Google Pixel~6a
(Tensor~G1 SoC). While the non-monotonic throughput ordering replicates on x86 AVX2
(Intel i5-1235U) and the Qwen~2.5~1.5B generalizability check confirms model-independence,
we have not tested Snapdragon~8~Gen~3 or MediaTek~Dimensity ARM cores.
The Cortex-X1's 512~KB L2 sets the ARM cliff threshold at ctx=512 (matching $L2/1024$ exactly,
confirmed by n=10 trials); Snapdragon's Cortex-X4 has a larger L2 and would exhibit a later cliff
predictable by the same formula.

\textbf{Quality evaluation scope.} All six benchmarks (BoolQ, ARC-Easy, ARC-Challenge,
HellaSwag, MMLU, TruthfulQA) are evaluated at n=100 questions per variant on both
ARM and x86 platforms. The 100-question sample yields Wilson 95\% CI of
$\approx\pm9.6\%$; fine-grained orderings that fall within one CI width should
be treated as directional rather than definitive.
Remaining gaps include GSM8K (math reasoning) and HumanEval (code generation),
which would extend quality coverage to generative tasks beyond MCQ.
WikiText-2 perplexity was measured on the full 285K-token test set
(568 chunks, \texttt{llama-perplexity} on-device), confirming the
PPL ordering is stable across the complete corpus.
The non-monotonic quality ordering (more bits $\neq$ better accuracy) warrants
investigation across additional model families and quantization frameworks.

\textbf{Working memory measurement only.} The peak memory figures in Figure~\ref{fig:memory}
reflect the MemAvailable delta in \texttt{/proc/meminfo}---i.e., the KV-cache plus
activation working memory---not the full RSS including memory-mapped model weights.
The model weights themselves contribute an additional 1.3--3.4~GB depending on
variant, loaded via mmap.

\textbf{Battery measurement granularity.} The benchmark was conducted over WiFi ADB
with the device unplugged, draining from 78\% to ${\sim}$2\%
(${\sim}$2.1\%/1k~tokens across ${\sim}$35{,}000 output tokens).
Android's \texttt{dumpsys battery} reports only integer-percentage values; each
30-second trial consumes $<$1\%, making per-trial tracking noise-dominated
(63 of 277 trials show a 1\% drop, the rest 0\%). Per-variant energy comparison
requires a hardware power monitor (e.g., Monsoon Solutions) or
longer-duration trials ($\geq$256 output tokens).

\textbf{Thermal trajectory characterized.}
We ran a dedicated thermal drift experiment: 5 baseline trials of Q4\_K\_M
at ctx=256 with a short prompt and 64 output tokens (fresh-context, no cooldown),
followed by 50 consecutive back-to-back trials (no inter-trial pause), then 5
recovery trials after a 140-second cooldown.
Baseline fresh-context throughput was $8.33\pm0.58$~tok/s.
Note: this baseline is higher than the canonical filled-context measurement
($4.78$~tok/s, Table~\ref{tab:main_results}) because with a short input prompt
the KV cache grows only from $\sim$20 to $\sim$84 tokens during decode, whereas
filled-context attention starts from 192 tokens and is proportionally more expensive.
Under sustained no-cooldown load, TPS dropped to a steady-state plateau of
$4.72$--$4.96$~tok/s within the first 10 trials (elapsed $<$65~seconds) and
remained stable for the next 900~seconds ($\sigma=0.07$).
After the 140-second recovery, TPS partially restored to $7.04\pm0.30$~tok/s.
This confirms two points: (1) the Tensor~G1 throttles under sustained CPU load
within $\approx$60~seconds, reducing fresh-context throughput $\approx40\%$;
(2) steady-state throughput is \emph{stable}---the $\pm$0.07~tok/s noise floor
after throttling onset indicates thermal throttling is consistent, not drifting.
All primary benchmarks in this paper use a 120-second inter-experiment cooldown,
which is sufficient to ensure thermally-stable (not cold-start) conditions.
The relative variant ordering reported in Table~\ref{tab:main_results} is
unaffected by thermal state, as all variants were measured under the same cooldown
protocol.

\subsection{Future Work}

\textbf{Imatrix calibration for mid-range variants.}
Importance-matrix-calibrated re-quantization via \texttt{llama-imatrix} + \texttt{llama-quantize}
is tractable (${\approx}$45~min for 200 calibration chunks on CPU) and can improve
PPL by 2--6\% at 4--5 bits.
We ran two independent BoolQ evaluations (n=100 each) for Q2\_K-imatrix and Q3\_K\_M-imatrix
on the Pixel~6a: Q2\_K degraded $-4$ to $-5$~pp (65\%~vs~69\% and 64\%~vs~69\% respectively),
Q3\_K\_M degraded $-7$ to $-8$~pp (62\%~vs~69\% and 61\%~vs~69\%).
A TruthfulQA evaluation shows no degradation ($\pm$1~pp: Q2\_K 51\%~vs~50\%, Q3\_K\_M 68\%~vs~68\%).
While the BoolQ decrements do not individually reach statistical significance (Wilson CI $\approx\pm$9.6~pp at n=100),
the consistent direction across two independent runs suggests a real negative effect
in the $-4$ to $-8$~pp range for sub-4-bpw variants.
This is consistent with the literature finding that calibration benefit diminishes
sharply below 3~bpw and that the Q2\_K HellaSwag collapse is a regime failure
rather than a weight-precision problem addressable by imatrix.
The most tractable target is Q4\_K\_S ($\geq$4~bpw), where imatrix shows neutral-to-positive
effects and may further widen the accuracy lead over Q4\_K\_M.

\textbf{Cross-platform comparison.} Repeating this benchmark on Snapdragon 8 Gen 3,
Apple A-series, and MediaTek Dimensity platforms would establish whether the
non-monotonic throughput finding is universal across ARM GGUF inference or
specific to the Tensor G1's memory subsystem.

\textbf{GPU and NPU acceleration.} The Tensor G1 includes an Arm Mali-G78 GPU
and a Google TPU block. Integrating these via OpenCL or delegating to TFLite's
NNAPI backend could dramatically change the efficiency--quality Pareto frontier.

\textbf{Context lengths beyond 2048.}
Q2\_K's cliff begins at ctx=512 and reaches $-48\%$ by ctx=2048.
Whether Q4\_K\_M and Q8\_0 (both stable to ctx=2048 in this study) remain
stable at 4096 and 8192 tokens---where KV-cache growth represents a larger
fraction of per-step DRAM traffic---is an open question predictable by
$L2/1024$ if per-thread L2 allocation is known for those systems.

\textbf{Qwen 2.5 1.5B cliff characterization.}
The empirical formula $\text{ctx}_{\text{cliff}} = L2/1024$ predicts ctx=512 for Qwen~2.5~1.5B
on ARM (same L2=512~KB), which matches the observed Qwen cliff onset in our n=5 sweep.
Notably, despite Qwen's different KV head count (2 KV heads vs.\ Llama's 8), the
same formula applies, suggesting the 1024 bytes/token empirical constant is more
robust to model architecture than the raw per-layer K+V formula.
A dedicated sequential sweep at $n{\geq}10$ trials per context point would
further strengthen the model-independence claim.

\textbf{Alternative frameworks.}
MLC-LLM~\cite{mlcllm} is not a viable apples-to-apples comparator for this study:
its Android runtime is GPU-only (OpenCL/Vulkan) with no ARM NEON CPU path,
and it has no Q2, Q5, Q6, or Q8 format equivalents.
The closest viable candidates to isolate whether the Q6\_K anomaly is
llama.cpp-specific are MNN-LLM (explicit ARM CPU NEON/i8mm path)
and ExecuTorch~\cite{executorch} with XNNPACK backend, which would test the
same SIMD dequantization question under a different kernel implementation.

\textbf{KV-cache quantization eliminates the Q2\_K cliff but reduces peak throughput.}
We evaluated \texttt{-ctk q8\_0 -ctv q8\_0} (8-bit KV cache, halving the fp16 footprint)
on Q2\_K, Q3\_K\_M, and Q4\_K\_M using the filled-context methodology.
Results are striking: Q2\_K's $-51\%$ cliff (7.50 to 3.69~tok/s, 256$\to$2048) is
\textbf{eliminated} by KV Q8\_0---throughput drops only $-2.6\%$ (4.04 to 3.93~tok/s),
flat across all contexts.
However, KV Q8\_0 reduces Q2\_K peak throughput by $\mathbf{-46\%}$ at ctx=256
(7.50 $\to$ 4.04~tok/s), because the attention kernel must now dequantize Q8\_0 KV
entries on every decode step---adding fixed overhead regardless of context length.
The crossover where KV Q8\_0 becomes beneficial for Q2\_K is around ctx=1400 tokens
(3.75~tok/s default vs.\ 4.00~tok/s with KV Q8\_0).
For Q3\_K\_M ($-11\%$ cliff) and Q4\_K\_M ($-12\%$ cliff), KV Q8\_0 roughly halves
the degradation ($-5\%$) with only a $\sim$1\% throughput penalty at ctx=256---a
favorable trade for applications targeting ctx$\geq$1024.
\textbf{Recommendation}: enable KV Q8\_0 when ctx$\geq$1400 with Q2\_K, or when
using Q3\_K\_M/Q4\_K\_M at ctx$\geq$1024 for modest but consistent improvement.

\textbf{Speculative decoding with a draft model.} Meta's Llama 3.2 1B Instruct
provides a natural draft model for speculative decoding~\cite{leviathan_speculative}
with the Llama 3.2 3B verifier. Combined GGUF size (Q4\_K\_M 3B + 1B) is ${\approx}2.7$\,GB,
fitting within the Tensor G1's 6\,GB budget. At the memory-bandwidth-saturated
decode regime observed here, CPU speculative decoding may yield 1.2--1.8$\times$
speedup with a suitable calibration corpus for the draft model.

\textbf{P-core CPU affinity.}
llama.cpp supports \texttt{--cpu-mask} to pin inference threads to specific cores.
On the Tensor~G1, the correct performance-core mask is \texttt{0xF0}
(cores~4--7: 2$\times$Cortex-X1 at 2.8~GHz + 2$\times$Cortex-A76 at 2.25~GHz);
mask \texttt{0x0F} targets the little cores (cores~0--3, Cortex-A55 at 1.8~GHz)
and reduces throughput by $-25\%$ for Q2\_K.
Measuring \texttt{0xF0} vs.\ the default scheduler is an immediately tractable
experiment; community data on similar SoCs suggests 10--25\% improvement.

\textbf{Hardware performance counter validation.}
The mechanistic analysis in this paper is based on static instruction-count
analysis of llama.cpp NEON/AVX2 source code.
Direct validation via ARM Performance Monitor Unit (PMU) counters---specifically
L2 cache miss rate, NEON utilization, and stall cycles per token---would
confirm whether the Q6\_K throughput penalty is dispatch-bound (many short
vector operations) or bandwidth-bound (frequent L2 misses from larger weight
tables).
Android's \texttt{simpleperf} tool can expose ARM PMU counters on non-rooted
devices for profiling on the Pixel~6a.

\textbf{Broader quality evaluation.}
All six benchmarks in this study (BoolQ, ARC-Easy, ARC-Challenge, HellaSwag,
MMLU, TruthfulQA) are classification or multiple-choice tasks; the model selects
among a fixed answer set.
Extending evaluation to generative tasks---GSM8K (grade-school math, chain-of-thought
reasoning) and HumanEval (Python code generation and execution)---would test
whether the Q2\_K HellaSwag collapse (near-random on MCQ) extends to open-ended
generation, where degenerate output modes may be harder to detect.
Based on the observed Q2\_K TruthfulQA failure (50\% on yes/no questions = chance)
and HellaSwag collapse (19\%, below chance), we hypothesize that Q2\_K will
show severe degradation on GSM8K and HumanEval as well~\cite{li_eval}.
Preliminary on-device evaluation was attempted but revealed a binary interaction
issue between \texttt{llama-completion}'s interactive mode and file-based prompt
input; results from this attempt are excluded.
Fixing the evaluation pipeline and running 50 GSM8K + 50 HumanEval questions
per variant is the most impactful remaining quality experiment.

\section{Conclusion}
% ===========================================================================

On CPU endpoints, the standard GPU intuition---higher bit-width yields higher
throughput---is wrong across every ISA we tested.
We identified three non-obvious phenomena, explained their root causes, and derived
a predictive model validated across 1,200+ measurements on ARM, x86, and GPU.

\textbf{Non-monotonic throughput (CPU-general).}
Q2\_K is fastest on both ARM (7.49~tok/s) and x86 (14.1~tok/s);
Q6\_K is slowest (3.53 and 6.8~tok/s)---a $2.1\times$ intra-format spread in
the wrong direction, reproduced on Qwen~2.5~1.5B.
The root cause is SIMD dequantization overhead: Q6\_K's split-bit layout
requires $\approx$26 NEON operations per block versus $\approx$10 for Q2\_K.

\textbf{CPU/GPU divide.}
Apple M4 Metal reverses the ordering entirely (Q4\_K\_S fastest, Q8\_0 slowest),
confirming the non-monotonicity is a CPU property.
M4 CPU-only inference outperforms Metal for Q8\_0 ($1.97\times$) and Q6\_K,
because Metal's 4-bit dispatch paths eliminate the dequantization bottleneck
that defines CPU throughput ordering.

\textbf{KV-cache cliff with predictive formula.}
Q2\_K throughput collapses $-48\%$ on ARM (ctx=512) and $-50\%$ on x86
(ctx=1300--1400).
The threshold is predicted within 5\% on both platforms by $L2/1024$---the
context at which the per-layer combined K+V working set overflows L2.
Q3\_K\_M is immune because its higher FFN compute cost masks the proportional
attention overhead.

\textbf{Practical guidance.}
Q6\_K is strictly dominated on CPU: slower than all 4-bit variants and no
accuracy advantage.
Q4\_K\_S gives the best quality (74\% BoolQ); Q4\_K\_M gives the best
throughput-stability balance.
Q2\_K is viable for throughput-limited deployment but fails structured
reasoning tasks (19\% HellaSwag).
KV-cache Q8\_0 quantization eliminates the Q2\_K cliff at a $-46\%$ short-context cost.

\section{Experimental Reproducibility}

\subsection{Build Configuration}

The llama.cpp library was compiled from commit \texttt{b1-1a29907} using the
Android NDK version 29.0.14206865. Compilation flags:
\begin{verbatim}
-O3 -march=armv8.2-a+dotprod+fp16
-DGGML_USE_CPU_HBM=OFF
-DLLAMA_BUILD_TESTS=OFF
\end{verbatim}

The resulting \texttt{llama-cli} binary was pushed to \texttt{/data/local/tmp/}
on the device and executed via ADB shell. The Android app was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a, using CMake 4.2.3 and Gradle 8.x.
All source code, benchmark scripts, and raw results are available at:
\begin{center}
\url{https://github.com/krisdcosta/291_EAI}\\
\smallskip
{\small Artifact commit: \texttt{df065f7} (2026-04-10)}
\end{center}
The repository includes all \texttt{.jsonl} trial data under \texttt{results/},
benchmark automation scripts under \texttt{scripts/bench/}, and the quality
evaluation pipeline under \texttt{scripts/eval/}. The PMU validation script
referenced in \S\ref{sec:nonmonotonic} is at \texttt{scripts/bench/pixel\_neon\_perf.sh}.
The verified ground-truth metrics table is at \texttt{VERIFIED\_METRICS\_MASTER\_TABLE.md}.

\subsection{Invocation Template}

Each trial was executed with the following command template,
where \texttt{<prompt>} is padded to exactly \texttt{ctx - 64} tokens
to saturate the KV cache (filled-context methodology):
\begin{verbatim}
./llama-cli \
  -m <model>.gguf \
  -p "<prompt>" \
  -n 64 \
  -c <ctx> \
  --no-mmap \
  -ngl 0 \
  -t <threads> \
  --log-format json \
  --seed <trial_seed>
\end{verbatim}

\subsection{Statistical Methodology}

Means and standard deviations in Table~\ref{tab:main_results} are computed
across $n{=}10$ trials per (variant, context) cell using the filled-context protocol
(prompt $= N - 64$ tokens, 128 output tokens).
Trial~1 is excluded as warmup; trials 2--11 are used.
Outlier rejection was not applied; all completed trials within the 600-second
timeout are included in analysis.

\section{Quality Evaluation Details}

\subsection{BoolQ (100 questions)}
BoolQ questions are drawn from the SuperGLUE validation split~\cite{boolq}.
Each example consists of a passage (avg.\ $\sim$100 words) and a yes/no question.
The prompt template is:
\begin{verbatim}
Passage: {passage}
Question: {question}
Answer with only yes or no:
\end{verbatim}
Responses are scored by extracting the first occurrence of the words
\emph{yes} or \emph{no} (case-insensitive) from the model output,
within 400 characters following the instruction marker.

\subsection{ARC-Easy (100 questions)}
ARC-Easy questions are drawn from the AI2 ARC Easy test split~\cite{arc}.
Each example presents a 4-choice multiple-choice question.
The prompt template is:
\begin{verbatim}
Question: {question}
Choices: A) {A} B) {B} C) {C} D) {D}
Answer with only A, B, C, or D:
\end{verbatim}
Scoring uses exact-match on the predicted letter (A/B/C/D).
The \texttt{model\_output} field in stored JSON records the extracted letter only
(e.g., ``A''), not the raw llama-completion output, ensuring stored results are
directly verifiable and reproducible. Verified accuracy across all seven variants:
Q2\_K 76\%, Q3\_K\_M 78\%, Q4\_K\_S 81\%, Q4\_K\_M 82\%, Q5\_K\_M 81\%, Q6\_K 79\%, Q8\_0 80\%.
