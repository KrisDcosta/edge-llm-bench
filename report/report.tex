On-Device Inference Efficiency for Edge AI Deployment}

\author{
\IEEEauthorblockN{Kris D'Costa}

\textit{University of California, San Diego}\\
San Diego, CA, USA\\

\begin{abstract}
We present a systematic empirical study of six GGUF quantization variants of
Llama~3.2~3B Instruct on a Google Pixel~6a (Tensor G2, 6~GB LPDDR5) using
llama.cpp, yielding 661 valid measurements across five quantization levels, four
context lengths (256--2048 tokens), and a thread-count sweep.
We find four non-obvious results.
(1)~\textbf{Throughput is non-monotonic with bit-width}: Q2\_K leads at 5.66~tok/s
while Q6\_K is \emph{slowest} at 3.98~tok/s, 20\% below Q8\_0, due to kernel
de-quantization overhead on ARM NEON.
(2)~\textbf{Context-length sensitivity is variant-dependent}: Q4\_K\_M ($+6\%$)
and Q2\_K ($-4\%$) are stable to 2048 tokens; Q8\_0 declines mildly ($-11\%$);
while Q3\_K\_M drops 38\% and Q6\_K drops 49\% at ctx=2048, exposing a hidden
memory-system fragility in these two quantisation schemes.
(3)~\textbf{Accuracy is also non-monotonic}: on a 100-question BoolQ reading
comprehension benchmark, Q4\_K\_M (72\%, 95\% CI: 63--80\%) achieves the highest
accuracy while Q6\_K scores lowest (65\%, CI: 55--74\%), despite having more bits
per weight. On 100-question ARC-Easy factual retrieval, all quantized variants score
95--100\% (near-ceiling), confirming the BoolQ result as the primary quality discriminator.
(4)~\textbf{Q6\_K is strictly dominated}: simultaneously the least accurate
and the slowest at long context; the Pareto frontier is defined by Q4\_K\_M
(accuracy-dominant, 72\%, 5.32~tok/s) and Q2\_K (speed-dominant, 69\%, 5.66~tok/s).
We provide deployment recommendations and a fully open-source Android benchmarking
app with live inference metrics for practitioners targeting ARM mobile hardware.
\end{abstract}

\section{Introduction}
% ===========================================================================

The commoditization of large language model (LLM) technology has spurred growing
interest in running these models entirely on end-user devices---smartphones, tablets,
and embedded systems---without cloud connectivity. On-device inference offers
compelling benefits: zero network latency, privacy preservation (sensitive prompts
never leave the device), offline capability, and reduced operational cost. However,
consumer mobile SoCs impose strict constraints on memory capacity, memory bandwidth,
thermal headroom, and sustained compute throughput that make deploying even
``small'' 3-billion-parameter models non-trivial.

Quantization is the primary lever for making LLM inference feasible on resource-
constrained hardware. By reducing the numerical precision of model weights from
the training-time 16-bit floating point to 4-bit or even 2-bit integer
representations, quantization can reduce model size by $4\times$--$8\times$ and
simultaneously improve memory-bandwidth utilization on hardware that lacks
dedicated matrix-multiplication engines for low-precision arithmetic. The GGUF
format~\cite{gguf}, popularized by the llama.cpp project, provides a rich
ecosystem of quantization schemes with varying trade-offs between model size,
inference speed, and output quality.

Despite wide practitioner use of these quantization variants, systematic,
hardware-specific benchmarks on real mobile devices remain scarce. Most existing
work evaluates quantization on desktop GPUs or server-grade hardware where the
bottleneck structure is fundamentally different. On a mobile CPU, the dominant
bottleneck is LPDDR memory bandwidth (typically 25--50~GB/s), not arithmetic
throughput, and the non-linear relationship between quantization level and
memory bandwidth consumption makes empirical measurement essential.

\textbf{Research Questions.} This paper addresses five concrete research questions:
\begin{itemize}
  \item \textbf{RQ1:} How does quantization level affect decode throughput, prefill
        throughput, time-to-first-token (TTFT), and end-to-end latency on a
        real mobile ARM SoC?
  \item \textbf{RQ2:} Does decode throughput degrade as context length grows from
        256 to 2048 tokens (i.e., does KV-cache growth impact performance)?
  \item \textbf{RQ3:} What fraction of total inference time is consumed by prefill
        versus decode, and how does this split vary across quantization levels?
  \item \textbf{RQ4:} What is the efficiency--accuracy trade-off across variants,
        measured by on-device BoolQ reading comprehension and ARC-Easy accuracy?
  \item \textbf{RQ5:} What are the practical deployment limits of each variant
        on a 6~GB LPDDR5 device, including thread-count scaling?
\end{itemize}

\textbf{Contributions.} We make the following contributions:
\begin{enumerate}
  \item A rigorous benchmark of six GGUF quantization variants of Llama~3.2~3B
        Instruct on physical Android hardware, yielding 661 valid measurements
        across four context lengths and a thread-count sweep.
  \item Discovery and explanation of the non-monotonic decode throughput anomaly
        in which Q6\_K (6-bit) underperforms both Q8\_0 (8-bit) and Q4\_K\_M
        (4-bit) due to kernel de-quantization overhead.
  \item Discovery that context-length sensitivity is \emph{variant-specific}:
        Q4\_K\_M and Q2\_K are context-stable ($<$5\% across 256--2048 tokens);
        Q8\_0 shows mild $-11\%$ decline; while Q3\_K\_M and Q6\_K collapse by
        38\% and 49\% at ctx=2048---a hidden fragility with significant deployment
        implications.
  \item A thread-count scaling study on the Tensor G2's heterogeneous ARM cluster
        showing a 5.0~tokens/s peak at 4 threads with degradation at 8.
  \item Complete on-device quality evaluation across all six variants using BoolQ
        (100-question reading comprehension) and ARC-Easy (100-question factual
        retrieval), revealing a non-monotonic BoolQ accuracy pattern (65--72\%):
        Q4\_K\_M scores highest at 72\%, Q6\_K lowest at 65\% despite its higher
        bit-width; Q6\_K is strictly dominated on both BoolQ accuracy and throughput.
        ARC-Easy is near-ceiling (95--100\%) for all variants.
\end{enumerate}

\section{Related Work}
% ===========================================================================

\subsection{LLM Quantization}

Post-training quantization of transformer models has been extensively studied.
GPTQ~\cite{gptq} introduced second-order weight quantization, achieving near-lossless
4-bit compression for large models. AWQ~\cite{awq} improved upon this by identifying
and protecting salient weight channels. SmoothQuant~\cite{smoothquant} tackled
activation quantization by migrating quantization difficulty from activations to
weights through mathematically equivalent channel-wise scaling.

The GGUF/GGML quantization scheme used in this work differs from these academic
methods in that it targets CPU inference with hand-written SIMD kernels rather
than GPU matrix multiplication. Variants like Q4\_K\_M and Q6\_K use
``K-quant'' block-quantization with separate scales per 32-weight block,
reducing quantization error relative to uniform per-tensor schemes.

\subsection{Mobile and Edge LLM Inference}

MLC-LLM~\cite{mlcllm} demonstrated that compilation-based optimization can
achieve competitive LLM inference on mobile GPUs, including Adreno and Apple
Neural Engine backends. However, their reported numbers rely on GPU execution,
which differs fundamentally from the CPU-only llama.cpp path evaluated here.

MediaTek and Qualcomm have published internal benchmarks for on-device LLM
inference using their respective NPU backends~\cite{qualcomm_llm}. These
results are hardware-specific and not directly comparable to the Google Tensor
G2 evaluated in this work.

EdgeMoE~\cite{edgeMoE} explored mixture-of-experts routing for edge inference,
while PowerInfer~\cite{powerinfer} proposed CPU/GPU split execution leveraging
neuron activation sparsity. Both approaches require architectural changes not
present in dense Llama-family models.

\subsection{llama.cpp on Mobile}

The llama.cpp project~\cite{llamacpp} supports Android via the NDK and
compiles NEON SIMD kernels for arm64-v8a targets. Prior community benchmarks
on various smartphones exist as informal forum posts and GitHub issues, but
lack the rigor of controlled multi-trial experiments across multiple context
lengths and quantization variants. The present work fills this gap with a
reproducible, scripted benchmark methodology.

\section{Methodology}\label{sec:methodology}
% ===========================================================================

\subsection{Target Platform}

All experiments were conducted on a \textbf{Google Pixel 6a} running Android 16
(API level 36). The device features the \textbf{Google Tensor G2} SoC, which
contains a tri-cluster ARM CPU configuration: 2$\times$ Cortex-X1 big cores,
2$\times$ Cortex-A76 mid cores, and 4$\times$ Cortex-A55 little cores. The
device ships with \textbf{6~GB LPDDR5} RAM. The LPDDR5 interface on the Tensor G2
provides a theoretical peak bandwidth of approximately 51.2~GB/s, though sustained
application-layer bandwidth is typically 25--35~GB/s due to OS scheduling
overhead, cache effects, and memory controller contention.

\subsection{Model and Quantization Variants}

The model under evaluation is \textbf{Llama 3.2 3B Instruct}~\cite{llama32},
Meta AI's 3-billion-parameter instruction-tuned model. We evaluate six quantization
variants available in the GGUF format~\cite{gguf}, spanning the full range from
2-bit to 16-bit precision:

\begin{itemize}
  \item \textbf{Q2\_K}: 2-bit K-quant, 1.3~GB on disk.
  \item \textbf{Q3\_K\_M}: 3-bit K-quant (medium), 1.6~GB.
  \item \textbf{Q4\_K\_M}: 4-bit K-quant (medium), 2.0~GB. Community standard.
  \item \textbf{Q6\_K}: 6-bit K-quant, 2.7~GB. Near-lossless compression.
  \item \textbf{Q8\_0}: 8-bit quantization, 3.4~GB. Heaviest quantized variant.
  \item \textbf{F16}: 16-bit floating point, 6.4~GB. Full reference precision.
\end{itemize}

\subsection{Experiment Design}

We adopt a fully-crossed factorial design across four independent variables:
quantization variant (6 levels), context length (256, 512, 1024, 2048 tokens),
prompt type (short factual, medium summarization, reasoning task), and trial
(5 runs per combination). We additionally include a thread-count sweep
(1, 2, 4, 8 threads) for Q4\_K\_M at 256-token context, and a dedicated
extended-context sweep (ctx=2048) for all five non-F16 variants. In total, the
experiment produced \textbf{661 successful measurements} after discarding warmup
runs, timed-out F16 trials, and one contaminated Q8\_0 ctx=2048 run
(discarded due to concurrent-process memory interference; replaced by a
clean sequential re-run). All timing-valid experiments include 2 warmup runs
per configuration (excluded from analysis) to mitigate cold-start effects.

\subsection{Metrics}

We measure the following metrics, extracted from llama-cli's structured timing output:
\begin{itemize}
  \item \textbf{Decode throughput} (tokens/s): Rate of output token generation.
        Primary metric for interactive user experience.
  \item \textbf{Prefill throughput} (tokens/s): Rate of input prompt processing.
  \item \textbf{Time-to-first-token (TTFT)} (s): Wall-clock time to first output
        token. Directly impacts perceived responsiveness.
  \item \textbf{End-to-end latency} (s): Total time for prefill + decode of 128
        output tokens.
  \item \textbf{BoolQ accuracy} (\%): Exact-match accuracy on a 100-question
        BoolQ reading comprehension benchmark (yes/no questions with passages).
  \item \textbf{ARC-Easy accuracy} (\%): 4-choice accuracy on a 100-question
        ARC-Easy elementary science benchmark.
\end{itemize}

\subsection{Quality Evaluation}

To characterize the efficiency--accuracy trade-off (RQ4), we conducted two
standard benchmark evaluations directly on-device via USB ADB for all six variants
(including F16).

\textbf{BoolQ (100 questions):} Reading comprehension benchmark~\cite{boolq}
drawn from the SuperGLUE validation split. Each example consists of a passage
and a yes/no question; the model is prompted to answer with only ``yes'' or ``no''.
Responses are scored via case-insensitive first-match extraction of the words
\emph{yes} or \emph{no} from the model output. The 100-question subset yields
a 95\% Wilson CI of $\pm$8--9\%, sufficient to distinguish variants that differ
by $>$7\%.

\textbf{ARC-Easy (100 questions):} Elementary science 4-choice benchmark~\cite{arc}
from the AI2 ARC Easy test split. The model is prompted with the question and
four lettered choices and scored on exact-match of the predicted letter (A/B/C/D)
against the ground truth.

\subsection{Repeatability Verification}

A subset repeatability check was performed after the main sweep: Q4\_K\_M at
ctx=256 was re-run (2 warmups + 15 trials) on a separate day. The
summarize and reasoning prompt decode rates matched the original dataset within
5.6\% ($5.47 \pm 0.12$ vs.\ $5.18 \pm 0.06$~tokens/s), which is within the
expected day-to-day variability of mobile hardware ($\pm$10--15\%) due to
OS scheduling and thermal starting conditions. One trial exhibited a transient
drop to 3.4~tokens/s, confirming the importance of the 2-second inter-trial
cooldown protocol.

\subsection{Framework Selection and Build}

We selected llama.cpp~\cite{llamacpp} as the inference backend after evaluating
ExecuTorch~\cite{executorch}: ExecuTorch's \texttt{LlmModule.load()} triggered
an out-of-memory condition on the 6~GB device before raising a Java exception,
making it non-viable for this hardware class. llama.cpp was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a with NEON SIMD kernels.
The llama-cli binary was pushed to the device via \texttt{adb push} and
invoked through WiFi ADB to avoid USB charging interference with thermal
measurements. The default thread count is 4 (matching the number of big+mid
cores: 2$\times$ X1 + 2$\times$ A76) unless otherwise stated.

Figure~\ref{fig:decode_tps} visualizes the decode throughput distribution.
Several secondary observations emerge:

\textbf{Q8\_0 achieves the best TTFT} at 3.95 seconds, tied with Q4\_K\_M (3.98~s)
and notably faster than Q2\_K (3.81~s). Despite Q8\_0's larger model size,
its kernel requires no de-quantization of scale factors during prefill, reducing
kernel dispatch overhead during the attention-intensive prefill phase.

\textbf{Q4\_K\_M achieves the highest BoolQ accuracy} (72\%, CI: 63--80\%) on
the 100-question reading comprehension evaluation. Q6\_K scores lowest at 65\%
(CI: 55--74\%), despite having more bits per weight than Q4\_K\_M. All quantized
variants score 95--100\% on ARC-Easy factual retrieval (near-ceiling), confirming
BoolQ as the primary discriminator among variants.

\textbf{F16 is effectively unusable} in this deployment scenario. At 0.15~tokens/s
with TTFT of 14.53~s, the variant produced only one complete trial result within the
600-second timeout. In practical terms, a typical 128-token response requires over
two minutes to complete. The F16 variant's 6.4~GB footprint also leaves minimal OS
headroom in a 6~GB device.

Context-length sensitivity is \textbf{variant-dependent}, not universal.
Table~\ref{tab:ctx2048} summarises decode throughput across all four context lengths.
Q4\_K\_M is strikingly context-invariant: 4.70, 5.03, 4.97, 5.26~tok/s across

Q2\_K is similarly stable ($-4\%$, 4.80 to 4.60~tok/s, well within noise).
Q8\_0 declines 11\% at ctx=2048 (4.74 to 4.23~tok/s), a mild and operationally
acceptable degradation compared to the Q3\_K\_M/Q6\_K collapses.
By contrast, Q3\_K\_M and Q6\_K show \textbf{severe performance collapse at
ctx=2048}: Q3\_K\_M drops from 4.46 to 2.76~tok/s ($-38\%$) and Q6\_K from
3.78 to 1.94~tok/s ($-49\%$) relative to ctx=1024.

This finding is mechanistically informative. The 2048-token KV-cache for
Llama~3.2~3B occupies $\approx$233~MB. For Q4\_K\_M (2.0~GB weights) this is
only 12\% of per-step memory traffic, keeping decode firmly weight-bandwidth-bound.
Q6\_K (2.7~GB weights) has an even smaller KV fraction (8.6\%) yet collapses---
pointing to Q6\_K's 6-bit asymmetric kernel imposing higher memory-access amplification
that becomes unstable when the KV footprint grows to 233~MB.
\textbf{Recommendation}: prefer Q4\_K\_M, Q2\_K, or Q8\_0 for
ctx$\geq$2048; avoid Q3\_K\_M and Q6\_K at maximum context.

Figure~\ref{fig:ttft_context} confirms that TTFT grows with context length
(as expected from the prefill phase) uniformly across all variants---the only
context-length-sensitive metric for the context-stable variants.

\subsection{RQ3: Prefill vs. Decode Time Fraction}
\label{sec:rq3}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig7_prefill_vs_decode_fraction.png}
  \caption{Fraction of total inference time consumed by prefill vs. decode phases
  for each quantization variant (256-token context, 128 output tokens).}
  \label{fig:prefill_decode_fraction}
\end{figure}

Figure~\ref{fig:prefill_decode_fraction} shows the time decomposition for 128-token
responses at 256-token context. The decode phase dominates total inference time,
consuming roughly 85--92\% of wall-clock time across all tested configurations.
The prefill fraction accounts for only 8--15\% of total latency.

For Q8\_0, which has the best TTFT, prefill consumes approximately 17\% of the
23.21~s end-to-end time. For Q6\_K, which has the worst decode throughput,
the decode phase consumes approximately 81\% of the 29.33~s total. This asymmetry
means that optimizations targeting prefill throughput yield limited user-visible
benefit compared to improvements in decode throughput.

\label{sec:rq4}

  \label{fig:pareto}
\end{figure}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig9_model_size_vs_decode_tps.png}
  \caption{Model size (GB) vs. decode throughput (tokens/s). Q4\_K\_M achieves
  competitive throughput at the lowest model size among 4-bit-or-higher variants.}
  \label{fig:size_vs_tps}

BoolQ reading comprehension accuracy was measured for all six variants via USB
ADB (100 questions per variant; 95\% Wilson CIs are shown in Table~\ref{tab:main_results}).
Q4\_K\_M leads at 72.0\% (72/100, CI: 63--80\%), followed by Q2\_K at 69.0\%
(69/100, CI: 59--77\%), Q8\_0 at 68.0\% (68/100, CI: 58--76\%), Q3\_K\_M at
66.0\% (66/100, CI: 56--75\%), and Q6\_K lowest at 65.0\% (65/100, CI: 55--74\%).
F16 also scores 68.0\% (68/100, CI: 58--76\%).
With $n{=}100$, the Wilson CIs are $\pm$8--9\%, making the 7-point gap between
Q4\_K\_M (72\%) and Q6\_K (65\%) the most statistically robust pairwise distinction.
The broader cluster between Q2\_K (69\%), Q8\_0 (68\%), and Q3\_K\_M (66\%) falls
within overlapping CIs and should be interpreted cautiously.
ARC-Easy accuracy was near-ceiling for all variants (100\% for Q2\_K through Q8\_0;
95\% for F16), confirming that these models retain strong factual retrieval capacity
at all quantization levels and that BoolQ---which requires cross-sentence reasoning---is
the more informative quality discriminator.

Figure~\ref{fig:pareto} plots the Pareto frontier for all variants.
\textbf{Two variants are Pareto-optimal on BoolQ accuracy vs.\ throughput}:
Q4\_K\_M (72\%, 5.32~tokens/s) is the accuracy-dominant point, while Q2\_K
(69\%, 5.66~tokens/s) is the speed-dominant point.
Notably, \textbf{Q8\_0 is no longer Pareto-optimal under BoolQ}: it scores 68\%
accuracy at 4.95~tokens/s, dominated by Q4\_K\_M which achieves both higher
accuracy \emph{and} higher throughput. Q3\_K\_M (66\%, 4.91~tokens/s) is also
dominated by Q4\_K\_M on both axes. \textbf{Q6\_K (65\%, 3.98~tokens/s) is
strictly dominated on both dimensions}---simultaneously the lowest-scoring and
slowest practical variant.

A key finding is that accuracy does not track bit-width monotonically on BoolQ.
Q4\_K\_M (4-bit) achieves the \emph{highest} accuracy, Q6\_K (6-bit)
scores \emph{lowest}---fewer correct answers than any other variant despite
having more bits per weight than Q4\_K\_M.
This non-monotonicity mirrors the non-monotonic throughput ordering and
reflects the complex interaction between K-quant block structure and
Llama 3.2's layer-specific weight sensitivity.

\textbf{Perplexity vs.\ BoolQ accuracy.}
WikiText-2 perplexity (Table~\ref{tab:main_results}) is nearly monotonic with
bit-width: Q8\_0 (9.70)~$\approx$~Q6\_K (9.75)~$\approx$~Q4\_K\_M (9.76)
$<$~Q3\_K\_M (10.15)~$<$~Q2\_K (11.71) (12\,KB sample, ${\approx}3{,}000$ tokens).
Yet Q6\_K and Q4\_K\_M have \emph{identical} PPL within 0.01 while their BoolQ
accuracy differs by 7\% (65\% vs.\ 72\%) and throughput by 34\%.
Perplexity is thus an \emph{insufficient} quality proxy: task-specific on-device
evaluation is essential to surface quantisation-specific failure modes.

\label{sec:rq5}

  \caption{Distribution of end-to-end latency across all trials for each quantization
  variant. Q3\_K\_M and Q6\_K exhibit notably tighter distributions than Q2\_K and Q8\_0.}
  \label{fig:latency_dist}
\end{figure}

\textbf{Thread-count scaling.} We measured Q4\_K\_M decode throughput at 1, 2, 4,
and 8 threads (256-token context, 3 trials per count). Results are:
1 thread: 2.8~tokens/s;
2 threads: 4.1~tokens/s;
4 threads: 5.0~tokens/s (peak);
8 threads: 4.0~tokens/s.

The Tensor G2's Cortex-X1 and A76 cores (4 total) drive the 4-thread peak.
At 8 threads, the 4 Cortex-A55 little cores are recruited, but their lower
clock speed and memory bandwidth share introduces load imbalance---the fast
big cores are often blocked waiting for little-core results---resulting in
a 20\% throughput \textit{regression} from 4 to 8 threads. This result
confirms that on heterogeneous ARM clusters, practitioners should not blindly
use all available cores; instead, thread count should match the number of
performance cores.

\textbf{Deployment feasibility by variant.}
Q2\_K (1.3~GB) and Q3\_K\_M (1.6~GB) are the most RAM-friendly and would
run on devices with as little as 3~GB total RAM. Q2\_K is speed-dominant
(5.66~tokens/s, 69\% BoolQ); Q3\_K\_M offers similar throughput (4.91~tokens/s)
at slightly lower accuracy (66\%), making Q2\_K the better overall choice
at this memory tier.
Q4\_K\_M (2.0~GB) comfortably fits any device $\geq$ 4~GB and is the recommended
default: it achieves the highest BoolQ accuracy (72\%) and second-highest
throughput (5.32~tokens/s), making it Pareto-dominant over Q8\_0 and Q3\_K\_M.
Q6\_K (2.7~GB) and Q8\_0 (3.4~GB) require $\geq$ 6~GB. Q8\_0 (68\%, 4.95~tokens/s)
is viable but dominated by Q4\_K\_M on both axes under BoolQ evaluation.
Q6\_K is the weakest operating point: at 65\% BoolQ accuracy and
only 3.98~tokens/s it is strictly dominated---prefer Q8\_0 or Q4\_K\_M on 6~GB
devices instead.
F16 (6.4~GB) is not viable on any consumer device $\leq$ 8~GB.

Figure~\ref{fig:latency_dist} shows latency distributions. Q3\_K\_M and Q6\_K
exhibit tighter distributions ($\sigma \leq 0.40$ tokens/s), indicating more
predictable behavior suitable for latency-SLA applications.

\section{Discussion}
% ===========================================================================

\subsection{The Non-Monotonic Throughput Anomaly}
\label{sec:nonmonotonic}

The most theoretically surprising result is that Q6\_K (6-bit, 2.7~GB) is slower
than both Q4\_K\_M (4-bit, 2.0~GB) and Q8\_0 (8-bit, 3.4~GB). On a naive
bandwidth model, higher bit-width means more bytes per weight and lower throughput.
The expected monotonic ordering would be Q2\_K $>$ Q3\_K\_M $>$ Q4\_K\_M $>$
Q6\_K $>$ Q8\_0. Our measured ordering is Q2\_K $>$ Q4\_K\_M $>$ Q8\_0 $\approx$
Q3\_K\_M $>$ Q6\_K.

We attribute the anomaly primarily to \textbf{kernel implementation overhead}
in the llama.cpp Q6\_K de-quantization path. The Q6\_K format packs 6-bit weights
using a combination of a 4-bit lookup table and 2-bit residuals, requiring
more complex de-packing logic per block compared to Q8\_0 (single 8-bit lookup)
or Q4\_K\_M (well-optimized 4-bit NEON kernel with vectorized scale application).
The de-quantization overhead partially offsets Q6\_K's memory bandwidth advantage,
yielding effective throughput below its bandwidth consumption would predict.

Similarly, Q3\_K\_M's moderate throughput (4.91~tokens/s despite 3-bit weights)
reflects the overhead of its medium-complexity block structure, which requires
scale factor lookups at finer granularity than Q4\_K\_M's implementation.

This finding highlights that the throughput ordering of quantization variants
is not determined by model size alone, but by the joint optimization of
memory bandwidth and kernel de-quantization efficiency. Practitioners should
treat Q6\_K with caution on ARM CPU backends.

\subsection{Memory Bandwidth as the Decode Bottleneck}

The context-length stability of Q4\_K\_M, Q2\_K, and Q8\_0 (Section~\ref{sec:rq2})
provides strong evidence that decode throughput on this device is
\textbf{memory-bandwidth saturated} by weight loading for these variants.
For Q4\_K\_M at 5.32~tokens/s, each decode step loads approximately 2.0~GB
of weights, implying an effective DRAM bandwidth utilization of
$2.0 \times 5.32 \approx 10.6$~GB/s for weight loading alone.
The KV-cache at 2048-token context contributes at most 233~MB per step,
less than 11\% of weight traffic---explaining Q4\_K\_M's near-zero throughput
variation across 256--2048 tokens. The Q3\_K\_M and Q6\_K collapses at ctx=2048
(Table~\ref{tab:ctx2048}) suggest that their quantisation kernels incur higher
effective memory-access amplification that makes them sensitive to the larger
allocation footprint when the KV-cache grows to 233~MB alongside these models.

\subsection{Deployment Recommendations}

\textbf{Default recommendation: Q4\_K\_M.} At 5.32 tokens/s, 2.0~GB, and 72\%
BoolQ accuracy with low variance (std~0.52), Q4\_K\_M provides the best overall
operating point for 6~GB devices. It is the accuracy-dominant Pareto point,
simultaneously outperforming Q8\_0 and Q3\_K\_M on both accuracy and throughput.
Q4\_K\_M's lower performance variance and broad community ecosystem support
further reinforce it as the pragmatic default.

\textbf{Speed-critical deployments: Q2\_K.} At 5.66~tokens/s and only 1.3~GB,
Q2\_K is the fastest practical option and is Pareto-optimal (69\% BoolQ, speed-dominant).
It is the best choice when throughput matters most and RAM is limited to 2~GB.

\textbf{Memory-constrained deployments (3--4~GB devices): Q2\_K or Q3\_K\_M.}
Both fit comfortably in 2~GB and can run on 3~GB devices. Q2\_K is preferred
(faster, slightly higher BoolQ accuracy). Q3\_K\_M is an alternative if the
66\% BoolQ accuracy is adequate and context stability at $\leq$1024 tokens
is acceptable.

\textbf{Avoid Q6\_K on CPU backends.} Q6\_K is strictly dominated in both
BoolQ accuracy (65\% vs.\ 66--72\% for all others) and throughput (3.98~tokens/s,
the slowest practical variant). Q4\_K\_M is faster, smaller, and 7\% more accurate;
Q8\_0 is more accurate and nearly 25\% faster. This recommendation may not apply
to GPU-accelerated backends where the Q6\_K de-quantization overhead differs.

\textbf{Use 4 threads on heterogeneous ARM.} On the Tensor G2 (and similar
big.LITTLE ARM SoCs), 4 threads matches the performance-core count and delivers
peak throughput. Do not use all available threads.

\textbf{Context length management (critical).} Context-length sensitivity is
\emph{variant-specific}. Q4\_K\_M ($+6\%$) and Q2\_K ($-4\%$) are practically
context-invariant through 2048 tokens and are the safest default for long-context
applications. Q8\_0 shows a mild $-11\%$ decline at ctx=2048, remaining operationally
viable. Q3\_K\_M
loses 38\% and Q6\_K loses 49\% of throughput at ctx=2048 relative to ctx=1024;
practitioners using these variants should limit context length to $\leq$1024
tokens or accept the throughput penalty. TTFT grows linearly with context across
all variants, so responsiveness decisions should be driven by TTFT budgets.

% ===========================================================================
\section{Limitations and Future Work}
\label{sec:limitations}
\label{sec:future}
% ===========================================================================

\subsection{Current Limitations}

\textbf{Single-device evaluation.} All results are from a single Google Pixel 6a.
The Tensor G2's specific characteristics may not generalize to Snapdragon,
MediaTek, or Apple ARM SoC families.

\textbf{Quality evaluation scope.} The 100-question BoolQ and ARC-Easy evaluations
cover reading comprehension and elementary factual recall; broader coverage
(MMLU, GSM8K, HumanEval, full WikiText-2 perplexity) would provide a more complete
quality characterization. The perplexity corpus used here (12\,KB, ${\approx}3{,}000$
tokens) is too small for statistically meaningful fine-grained PPL comparisons;
re-running with the full 285K-token WikiText-2 test set is recommended.
The non-monotonic BoolQ accuracy ordering warrants further investigation.

\textbf{Working memory measurement only.} The peak memory figures in Figure~\ref{fig:memory}
reflect the MemAvailable delta in \texttt{/proc/meminfo}---i.e., the KV-cache plus
activation working memory---not the full RSS including memory-mapped model weights.
The model weights themselves contribute an additional 1.3--3.4~GB depending on
variant, loaded via mmap.

\textbf{Battery measurement granularity.} The benchmark was conducted over WiFi ADB
with the device unplugged, draining from 78\% to ${\sim}$2\%
(${\sim}$2.1\%/1k~tokens across ${\sim}$35{,}000 output tokens).
Android's \texttt{dumpsys battery} reports only integer-percentage values; each
30-second trial consumes $<$1\%, making per-trial tracking noise-dominated
(63 of 277 trials show a 1\% drop, the rest 0\%). Per-variant energy comparison
requires a hardware power monitor (e.g., Monsoon Solutions) or
longer-duration trials ($\geq$256 output tokens).

\textbf{Thermal trajectory not characterized.} Despite a 120-second cooldown
between experiments, cumulative thermal state over the 3-hour benchmark session
may have affected later results. We report results in run order but do not
model thermal drift explicitly.

\subsection{Future Work}

\textbf{Full-corpus perplexity and imatrix calibration.}
Re-evaluating PPL on the full WikiText-2 test set (285\,K tokens) will validate
fine-grained comparisons. On-device benchmarking of importance-matrix-calibrated
re-quantizations (\texttt{Q2\_K}--\texttt{Q8\_0}) will assess whether calibration
narrows the accuracy gap at lower bit-widths.

\textbf{Cross-platform comparison.} Repeating this benchmark on Snapdragon 8 Gen 3,
Apple A-series, and MediaTek Dimensity platforms would establish whether the
non-monotonic throughput finding is universal across ARM GGUF inference or
specific to the Tensor G2's memory subsystem.

\textbf{GPU and NPU acceleration.} The Tensor G2 includes an Arm Mali-G710 GPU
and a Google TPU block. Integrating these via OpenCL or delegating to TFLite's
NNAPI backend could dramatically change the efficiency--quality Pareto frontier.

\textbf{Context lengths beyond 2048.} We found that Q3\_K\_M and Q6\_K collapse
at ctx=2048. Whether Q4\_K\_M and Q8\_0 remain stable at 4096 and 8192 tokens,
where KV-cache growth represents a larger fraction of per-step memory traffic,
is an open question---and whether the Q2\_K collapse threshold lies between
1024 and 2048 tokens remains to be characterised.

\textbf{Alternative frameworks.} Evaluating MLC-LLM~\cite{mlcllm} and a
future stable ExecuTorch~\cite{executorch} release on the same hardware would
clarify whether the Q6\_K anomaly is llama.cpp-specific or intrinsic to ARM NEON.

% ===========================================================================
\section{Conclusion}
% ===========================================================================

We have presented a rigorous, multi-factor benchmark of six GGUF quantization
variants of Llama 3.2 3B Instruct on a Google Pixel 6a (Tensor G2, 6~GB LPDDR5),
yielding 661 measurements across four context lengths (256--2048 tokens),
five quantization levels, and a thread-count scaling study.

Our primary empirical findings are:

\begin{enumerate}
  \item \textbf{Non-monotonic throughput}: Q6\_K (6-bit, 2.7~GB) delivers
        3.98~tokens/s---slower than Q4\_K\_M (5.32~tokens/s) and Q8\_0 (4.95~tokens/s)
        due to kernel de-quantization overhead on ARM NEON.
  \item \textbf{Context-length sensitivity is variant-specific}: Q4\_K\_M ($+6\%$)
        and Q2\_K ($-4\%$) are context-invariant through 2048 tokens; Q8\_0 shows
        a mild $-11\%$ decline. Q3\_K\_M and Q6\_K collapse at ctx=2048 ($-38\%$
        and $-49\%$ respectively), exposing a hidden memory-system fragility that
        makes them unsafe for maximum-context deployments.
  \item \textbf{Thread-count sweet spot}: 4 threads is optimal on the Tensor G2's
        heterogeneous cluster; 8 threads degrades throughput by 20\% due to
        little-core load imbalance.
  \item \textbf{BoolQ accuracy non-monotonicity}: On-device BoolQ reading comprehension
        accuracy does not monotonically improve with bit-width (65--72\%):
        Q4\_K\_M (4-bit) achieves the highest score at 72\%, Q6\_K (6-bit)
        scores lowest at 65\% despite having more bits per weight than Q4\_K\_M.
        ARC-Easy factual accuracy is near-ceiling (95--100\%) for all variants.
        Q6\_K is strictly dominated on both BoolQ accuracy and throughput,
        confirming it as the weakest deployment choice.
  \item \textbf{Q4\_K\_M recommended; Pareto-dominant on both accuracy and throughput}:
        The BoolQ Pareto frontier is defined by Q4\_K\_M (72\%, 5.32~tokens/s,
        accuracy-dominant) and Q2\_K (69\%, 5.66~tokens/s, speed-dominant).
        Q8\_0 is dominated by Q4\_K\_M on both axes under BoolQ. Q4\_K\_M
        (2.0~GB) offers the best overall balance for 6~GB deployment.
\end{enumerate}

These findings provide actionable guidance: use Q4\_K\_M as the default
(context-stable, highest BoolQ accuracy at 72\%, best overall balance),
Q2\_K for latency-constrained or RAM-limited deployments (fastest at 5.66~tok/s,
69\% BoolQ), 4 threads on big.LITTLE ARM, and avoid Q6\_K on CPU backends
entirely---it is strictly dominated on both BoolQ accuracy and throughput,
and catastrophically slow at ctx=2048. As on-device AI
continues to proliferate, hardware-specific benchmarks of this kind are essential
for evidence-based quantization selection.

\section{Experimental Reproducibility}

\subsection{Build Configuration}

The llama.cpp library was compiled from commit \texttt{b1-1a29907} using the
Android NDK version 29.0.14206865. Compilation flags:
\begin{verbatim}
-O3 -march=armv8.2-a+dotprod+fp16
-DGGML_USE_CPU_HBM=OFF
-DLLAMA_BUILD_TESTS=OFF
\end{verbatim}

The resulting \texttt{llama-cli} binary was pushed to \texttt{/data/local/tmp/}
on the device and executed via ADB shell. The Android app was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a, using CMake 4.2.3 and Gradle 8.x.
All source code, benchmark scripts, and raw results are available in the
project repository.

\subsection{Invocation Template}

Each trial was executed with the following command template:
\begin{verbatim}
./llama-cli \
  -m <model_path>.gguf \
  -p "<system_prompt><user_prompt>" \
  -n 128 \
  --ctx-size <context_length> \
  --seed <trial_seed> \
  -t <thread_count> \
  --log-format json
\end{verbatim}

\subsection{Statistical Methodology}

Means and standard deviations in Table~\ref{tab:main_results} are computed
across 15 samples per variant per context (5 trials $\times$ 3 prompt types).
Warmup runs (2 per configuration) are excluded. Outlier rejection was not applied;
all completed trials within the timeout threshold are included in analysis.

\section{Quality Evaluation Details}

\subsection{BoolQ (100 questions)}
BoolQ questions are drawn from the SuperGLUE validation split~\cite{boolq}.
Each example consists of a passage (avg.\ $\sim$100 words) and a yes/no question.
The prompt template is:
\begin{verbatim}
Passage: {passage}
Question: {question}
Answer with only yes or no:
\end{verbatim}
Responses are scored by extracting the first occurrence of the words
\emph{yes} or \emph{no} (case-insensitive) from the model output,
within 400 characters following the instruction marker.

\subsection{ARC-Easy (100 questions)}
ARC-Easy questions are drawn from the AI2 ARC Easy test split~\cite{arc}.
Each example presents a 4-choice multiple-choice question.
The prompt template is:
\begin{verbatim}
Question: {question}
Choices: A) {A} B) {B} C) {C} D) {D}
Answer with only A, B, C, or D:
\end{verbatim}
Scoring uses exact-match on the predicted letter (A/B/C/D).
