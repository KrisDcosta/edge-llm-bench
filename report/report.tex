\begin{abstract}
We present a systematic empirical study of seven GGUF quantization variants of
Llama~3.2~3B Instruct on a Google Pixel~6a (Tensor G1, 6~GB LPDDR5) using
llama.cpp, yielding 700+ valid measurements across throughput, context-scaling,
and quality benchmarks spanning four KV-cache context window sizes (256--2048 tokens)
and a thread-count sweep.
We find four non-obvious results.
(1)~\textbf{Throughput is non-monotonic with bit-width}: Q2\_K leads at 5.11~tok/s
while Q6\_K is \emph{slowest} at 3.52~tok/s, 22\% below Q8\_0, due to kernel
de-quantization overhead on ARM NEON.
(2)~\textbf{Context sensitivity is also non-monotonic}: using filled-context
sweeps (prompts filling the allocated KV window), Q2\_K degrades $-40\%$
(ctx=256$\to$2048, cliff onset at ctx=768) while Q3\_K\_M varies $<\pm5\%$
across all window sizes. Q4\_K\_M is also context-stable ($-7\%$), whereas
Q5\_K\_M ($-26\%$) and Q8\_0 ($-19\%$) degrade substantially.
This mirrors the throughput non-monotonicity: the fastest short-context variant
is the most fragile at long context.
(3)~\textbf{Accuracy is also non-monotonic across six benchmarks}:
on BoolQ reading comprehension, Q4\_K\_S leads (74\%, CI: 64--82\%) while Q6\_K
scores lowest (65\%); on TruthfulQA factual consistency, Q3\_K\_M leads (68\%) while
Q2\_K collapses to 50\% (near-chance). Q2\_K also exhibits instruction-following
collapse on HellaSwag MCQ (19\%), despite 69\% BoolQ accuracy---a task-format-specific
failure at 3.40~bpw. No single variant dominates all quality dimensions;
the optimal choice is benchmark-dependent.
(4)~\textbf{Q6\_K is strictly dominated}: simultaneously the least accurate
and the slowest at long context; Q4\_K\_S (74\%, 4.8~tok/s) achieves the highest
BoolQ accuracy, Q2\_K (69\%, 5.11~tok/s) leads on throughput, and Q4\_K\_M (72\%,
4.80~tok/s) provides the most context-stable operating point.
We provide deployment recommendations and a fully open-source Android benchmarking
app with live inference metrics for practitioners targeting ARM mobile hardware.
\end{abstract}

\section{Introduction}
% ===========================================================================

\subsection{Motivation: On-Device LLM Inference and the Quantization Imperative}

Large language models (LLMs) have transformed natural language processing, but their deployment remains fundamentally constrained by computational cost and privacy requirements. Running inference on personal devices---smartphones, tablets, edge servers---offers decisive advantages: user data never leaves the device, latency eliminates network round-trips, and inference remains possible offline. Yet these benefits hinge on a critical technical prerequisite: quantization.

Quantization compresses model weights and activations from floating-point (typically 32 or 16 bits) to lower precisions (2--8 bits), reducing memory footprint by 4--16$\times$ and enabling sub-gigabyte models to run on resource-constrained ARM processors. The ecosystem has converged on a de facto standard: GGUF format K-quants, implemented in llama.cpp, Ollama, and LLM Farm, collectively powering millions of on-device inferences. Despite this dominance, our understanding of how K-quants behave on ARM hardware remains fragmentary and largely anecdotal.

Existing quantization research is overwhelmingly GPU-centric. Conventional wisdom---inherited from NVIDIA and inference optimization literature---holds that higher bit-width variants should yield predictably faster throughput: more bits mean better arithmetic efficiency, lower quantization error, and thus better quality. Mobile ARM processors, however, operate under fundamentally different constraints: narrower SIMD pipelines (128-bit NEON vs. 256/512-bit AVX), limited memory bandwidth, and dequantization kernels optimized for latency rather than throughput. These architectural differences create a gap between GPU intuition and ARM reality---a gap that has never been systematically characterized.

\subsection{Problem Statement}

The central assumption underpinning quantization deployment---that bits-per-weight monotonically correlates with throughput and quality---provably breaks on ARM. Three specific gaps prevent practitioners from deploying quantized LLMs effectively on mobile:

\begin{itemize}
  \item \textbf{No characterization of KV-cache throughput sensitivity.} The key-value cache grows linearly with context window length, quickly dominating memory bandwidth on ARM. Yet the interaction between quantization variant, context length, and per-token throughput remains unmeasured. At what context window does a fast variant (e.g., Q2\_K) collapse? Does this threshold differ across quantization methods?

  \item \textbf{Variant-specific performance orderings are undocumented.} Which K-quant variant is actually fastest on ARM? Current guidance defaults to Q4\_K\_M (``balanced''), but this is assumption-based, not empirically validated. Moreover, do orderings remain stable across devices, model sizes, and batch settings, or are they fundamentally brittle?

  \item \textbf{imatrix calibration efficacy on real hardware is unknown.} Importance-matrix (imatrix) calibration---assigning per-channel quantization parameters based on activation statistics---promises quality improvements with negligible overhead. Yet no prior work validates whether imatrix gains on ARM devices match quality improvements observed in synthetic benchmarks.
\end{itemize}

\subsection{Contributions}

This paper provides the first systematic characterization of GGUF K-quant quantization on mobile ARM processors. We conduct 420+ controlled experiments on Pixel 6a (ARM Cortex-X1), measuring throughput, quality, and KV-cache behavior across 7 quantization variants and 4 models. Cross-device validation on additional ARM processors is left as future work. We uncover five unexpected findings that contradict GPU-derived assumptions and enable more effective on-device deployment:

\begin{description}
  \item[\textbf{RQ1: Non-monotonic throughput ordering.}] Across our ARM testbed, Q2\_K achieves 5.11 tokens/second---the fastest variant despite encoding only 3.40 bits per weight. Conversely, Q6\_K, which uses 6.59 bits per weight, achieves only 3.52 tokens/second. This 31\% throughput \emph{decrease} with higher precision directly contradicts GPU behavior and GPU-based quantization literature. Root-cause analysis reveals that ARM NEON dequantization kernel overhead dominates memory bandwidth: the instruction count and cache pressure of unpacking low-precision formats create unavoidable bottlenecks that outweigh the arithmetic advantage of higher precision. Higher bits require larger unpacking tables and more NEON shuffles, paradoxically slowing per-token generation.

  \item[\textbf{RQ2: Context sensitivity is non-monotonic with bit-width.}] Using a filled-context methodology (prompts sized to $N{-}64$ tokens to actually saturate the KV cache), we find that the fastest variant at short context (Q2\_K) is also the \emph{most} context-sensitive, degrading $-40\%$ decode TPS from ctx=256 to ctx=2048 with a cliff onset at ctx=768. Conversely, Q3\_K\_M (4.20 bpw) is remarkably context-stable, varying $<\pm5\%$ across all context windows. Q4\_K\_M similarly shows only $-6.6\%$ degradation. This non-monotonic pattern---where \emph{lower} bit-width correlates with \emph{higher} context sensitivity---mirrors the throughput ordering anomaly and reveals that ARM NEON dequantization overhead interacts with KV-cache memory pressure in a quantization-format-specific way.

  \item[\textbf{RQ3: Non-monotonic quality ordering contradicts bits-per-weight intuition.}] Q4\_K\_M (1.9~GB model footprint) outperforms Q6\_K (2.5~GB) on most measured benchmarks, despite having 2$\times$ fewer bits per weight. MMLU accuracy clusters tightly between 42--50\% across all variants (95\% Wilson CI width $\approx\pm10$pp), making pairwise distinctions statistically unreliable at $n=100$ questions. Q2\_K is detectably weaker (42\%), while Q3\_K\_M through Q8\_0 are statistically indistinguishable on MMLU alone. Q4\_K\_S (1.56~GB, 4.85~bpw) achieves the highest BoolQ accuracy at 74\%, outperforming both Q4\_K\_M (72\%) and Q6\_K (65\%). On TruthfulQA factual consistency (MC1), Q3\_K\_M leads at 68\% while Q2\_K collapses to 50\% (near chance for 4-choice), the same variant that is fastest (RQ1) and most context-fragile (RQ2). Root cause analysis reveals that K-quant superblock structure---in which per-channel scales and per-block importance weightings are coordinated---interacts non-trivially with model weight sensitivity at the 4--5 bit regime.

  \item[\textbf{RQ4: Cross-device portability of ARM NEON patterns.}] We characterize the extent to which the non-monotonic throughput ordering observed on Pixel 6a (Cortex-X1) generalizes to other backends. As a preliminary cross-device validation, we measured throughput for all 7 variants on a Mac M4 (Metal GPU backend). The ordering reverses substantially: Q4\_K\_S leads at 19.9~tok/s and Q4\_K\_M at 19.2~tok/s, while Q8\_0 (6.4~tok/s) and Q6\_K (7.0~tok/s) are slowest---the opposite of ARM, where Q8\_0 and Q6\_K are mid-pack and Q2\_K leads. Metal has well-tuned 4-bit kernels; 8-bit inference is not comparably optimized. Q2\_K, the ARM champion, is middle-pack on Metal (17.8~tok/s). This reversal underscores that mobile ARM is a distinct regime requiring dedicated characterization; GPU-based intuitions do not transfer.

  \item[\textbf{RQ5: imatrix calibration efficacy is variant-specific and generally minimal.}] On BoolQ, imatrix shows minimal improvement (maximum +4\% for Q6\_K), is ineffective at most precision levels ($\leq\pm1\%$ for Q4\_K\_M, Q5\_K\_M, Q8\_0), and is negative at low bit-widths ($-5\%$ for Q2\_K, $-5\%$ for Q3\_K\_M). MMLU and other benchmark evaluations of imatrix calibration were not performed due to resource constraints. These BoolQ results suggest that imatrix calibration provides limited practical benefit across the quantization spectrum and does not recover meaningful accuracy improvements.
\end{description}

\subsection{Roadmap}

The remainder of this paper is organized as follows. \S2 surveys related work on neural network quantization, mobile inference optimization, and recent advances in KV-cache compression. \S3 describes our experimental methodology, including hardware platforms, quantization variants, experiment design (420+ runs), and measurement protocol. \S4 presents our throughput results and analyzes the non-monotonic ordering phenomenon, isolating the ARM NEON kernel bottleneck. \S5 characterizes KV-cache collapse and derives predictive models for context-length thresholds. \S6 evaluates quality trade-offs and the role of imatrix calibration across precision levels. \S7 presents preliminary cross-device validation on Mac M4 Metal and discusses ordering reversals vs. ARM. \S8 synthesizes mechanistic explanations and discusses implications for deployment---including practical guidance for practitioners choosing quantization variants. \S9 concludes and outlines future directions in mobile-aware quantization research.

\section{Related Work}
% ===========================================================================

\subsection{LLM Quantization}

Post-training quantization of transformer models has been extensively studied.
GPTQ~\cite{gptq} introduced second-order weight quantization, achieving near-lossless
4-bit compression for large models. AWQ~\cite{awq} improved upon this by identifying
and protecting salient weight channels. SmoothQuant~\cite{smoothquant} tackled
activation quantization by migrating quantization difficulty from activations to
weights through mathematically equivalent channel-wise scaling.

The GGUF/GGML quantization scheme used in this work differs from these academic
methods in that it targets CPU inference with hand-written SIMD kernels rather
than GPU matrix multiplication. Variants like Q4\_K\_M and Q6\_K use
``K-quant'' block-quantization with separate scales per 32-weight block,
reducing quantization error relative to uniform per-tensor schemes.

\subsection{Mobile and Edge LLM Inference}

MLC-LLM~\cite{mlcllm} demonstrated that compilation-based optimization can
achieve competitive LLM inference on mobile GPUs, including Adreno and Apple
Neural Engine backends. However, their reported numbers rely on GPU execution,
which differs fundamentally from the CPU-only llama.cpp path evaluated here.

MediaTek and Qualcomm have published internal benchmarks for on-device LLM
inference using their respective NPU backends~\cite{qualcomm_llm}. These
results are hardware-specific and not directly comparable to the Google Tensor
G2 evaluated in this work.

EdgeMoE~\cite{edgeMoE} explored mixture-of-experts routing for edge inference,
while PowerInfer~\cite{powerinfer} proposed CPU/GPU split execution leveraging
neuron activation sparsity. Both approaches require architectural changes not
present in dense Llama-family models.

\subsection{llama.cpp on Mobile}

The llama.cpp project~\cite{llamacpp} supports Android via the NDK and
compiles NEON SIMD kernels for arm64-v8a targets. Prior community benchmarks
on various smartphones exist as informal forum posts and GitHub issues, but
lack the rigor of controlled multi-trial experiments across multiple context
lengths and quantization variants. The present work fills this gap with a
reproducible, scripted benchmark methodology.

\section{Methodology}\label{sec:methodology}
% ===========================================================================

\subsection{Target Platform}

All experiments were conducted on a \textbf{Google Pixel 6a} running Android 16
(API level 36). The device features the \textbf{Google Tensor G1} SoC, which
contains a tri-cluster ARM CPU configuration: 2$\times$ Cortex-X1 big cores,
2$\times$ Cortex-A76 mid cores, and 4$\times$ Cortex-A55 little cores. The
device ships with \textbf{6~GB LPDDR5} RAM. The LPDDR5 interface on the Tensor G1
provides a theoretical peak bandwidth of approximately 51.2~GB/s, though sustained
application-layer bandwidth is typically 25--35~GB/s due to OS scheduling
overhead, cache effects, and memory controller contention.

\subsection{Model and Quantization Variants}

The model under evaluation is \textbf{Llama 3.2 3B Instruct}~\cite{llama32},
Meta AI's 3-billion-parameter instruction-tuned model. We evaluate seven quantization
variants available in the GGUF format~\cite{gguf}, spanning the full range from
2-bit to 16-bit precision:

\begin{itemize}
  \item \textbf{Q2\_K}: 2-bit K-quant, 1.3~GB on disk (3.40~bpw).
  \item \textbf{Q3\_K\_M}: 3-bit K-quant (medium), 1.6~GB.
  \item \textbf{Q4\_K\_S}: 4-bit K-quant (small), 1.56~GB (4.85~bpw). Evaluated for quality benchmarks.
  \item \textbf{Q4\_K\_M}: 4-bit K-quant (medium), 2.0~GB. Community standard.
  \item \textbf{Q6\_K}: 6-bit K-quant, 2.7~GB (6.59~bpw). Near-lossless compression.
  \item \textbf{Q8\_0}: 8-bit quantization, 3.4~GB. Heaviest quantized variant.
  \item \textbf{F16}: 16-bit floating point, 6.4~GB. Full reference precision.
\end{itemize}

\subsection{Experiment Design}

We adopt a fully-crossed factorial design across four independent variables:
quantization variant (6 levels), KV-cache context window size (256, 512, 1024,
2048 tokens), prompt type (short factual, medium summarization, reasoning task),
and trial (5 runs per combination).

\textbf{Note on ``context length'':} In all experiments, the actual input prompts
contain 18--38 tokens. The parameter varied is the llama.cpp \texttt{-c} flag,
which sets the maximum \emph{KV-cache context window} allocated in memory. Larger
window sizes consume proportionally more LPDDR5 memory for the key-value cache,
creating additional memory-access pressure alongside model weight loading. Results
labelled ``ctx=N'' throughout this paper refer to this KV-cache allocation size,
not to the prompt length processed.

We additionally include a thread-count sweep
(1, 2, 4, 8 threads) for Q4\_K\_M at 256-token context, and a dedicated
extended-context sweep (KV-cache window=2048) for all six non-F16 variants. In total, the
experiment produced \textbf{700+ valid measurements across throughput, context-scaling,
and quality benchmarks} after discarding warmup runs, timed-out F16 trials, and one
contaminated Q8\_0 ctx=2048 run (discarded due to concurrent-process memory interference;
replaced by a clean sequential re-run). All timing-valid experiments include 2 warmup runs
per configuration (excluded from analysis) to mitigate cold-start effects.

\subsection{Metrics}

We measure the following metrics, extracted from llama-cli's structured timing output:
\begin{itemize}
  \item \textbf{Decode throughput} (tokens/s): Rate of output token generation.
        Primary metric for interactive user experience.
  \item \textbf{Prefill throughput} (tokens/s): Rate of input prompt processing.
  \item \textbf{Time-to-first-token (TTFT)} (s): Wall-clock time to first output
        token. Directly impacts perceived responsiveness.
  \item \textbf{End-to-end latency} (s): Total time for prefill + decode of 128
        output tokens.
  \item \textbf{BoolQ accuracy} (\%): Exact-match accuracy on a 100-question
        BoolQ reading comprehension benchmark (yes/no questions with passages).
  \item \textbf{ARC-Easy accuracy} (\%): 4-choice accuracy on a 100-question
        ARC-Easy elementary science benchmark.
\end{itemize}

\subsection{Quality Evaluation}

To characterize the efficiency--accuracy trade-off (RQ4), we conducted two
standard benchmark evaluations directly on-device via USB ADB for all seven variants
(including F16).

\textbf{BoolQ (100 questions):} Reading comprehension benchmark~\cite{boolq}
drawn from the SuperGLUE validation split. Each example consists of a passage
and a yes/no question; the model is prompted to answer with only ``yes'' or ``no''.
Responses are scored via case-insensitive first-match extraction of the words
\emph{yes} or \emph{no} from the model output. The 100-question subset yields
a 95\% Wilson CI of $\pm$8--9\%, sufficient to distinguish variants that differ
by $>$7\%.

\textbf{ARC-Easy (100 questions):} Elementary science 4-choice benchmark~\cite{arc}
from the AI2 ARC Easy test split. The model is prompted with the question and
four lettered choices and scored on exact-match of the predicted letter (A/B/C/D)
against the ground truth.

\subsection{Repeatability Verification}

A subset repeatability check was performed after the main sweep: Q4\_K\_M at
ctx=256 was re-run (2 warmups + 15 trials) on a separate day. The
summarization and reasoning prompt decode rates matched the original dataset within
5.6\% ($5.47 \pm 0.12$ vs.\ $5.18 \pm 0.06$~tokens/s), which is within the
expected day-to-day variability of mobile hardware ($\pm$10--15\%) due to
OS scheduling and thermal starting conditions. One trial exhibited a transient
drop to 3.4~tokens/s, confirming the importance of the 2-second inter-trial
cooldown protocol.

\subsection{Framework Selection and Build}

We selected llama.cpp~\cite{llamacpp} as the inference backend after evaluating
ExecuTorch~\cite{executorch}: ExecuTorch's \texttt{LlmModule.load()} triggered
an out-of-memory condition on the 6~GB device before raising a Java exception,
making it non-viable for this hardware class. llama.cpp was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a with NEON SIMD kernels.
The llama-cli binary was pushed to the device via \texttt{adb push} and
invoked through WiFi ADB to avoid USB charging interference with thermal
measurements. The default thread count is 4 (matching the number of big+mid
cores: 2$\times$ X1 + 2$\times$ A76) unless otherwise stated.

headroom in a 6~GB device.

\subsection{RQ2: Context-Length Scaling and KV-Cache Sensitivity}
\label{sec:rq2}

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{../figures/fig_kv_cliff.png}
  \caption{Decode throughput (tok/s) vs.\ context window for all seven variants under
  \emph{filled-context} conditions (input prompt sized to $N{-}64$ tokens so the
  KV cache is actually saturated at allocation $N$). Q2\_K collapses to 4.76~tok/s
  at ctx=2048 ($-40\%$); Q3\_K\_M stays flat ($<\pm5\%$).}
  \label{fig:kv_cliff}
\end{figure}

\textbf{Methodology note.} Prior work often sweeps context by varying the KV
allocation parameter~\texttt{-c N} while using a short fixed prompt.  This
measures only allocation overhead, not genuine KV-cache pressure.  We instead
generate prompts of $N{-}64$ tokens for each context size $N$, ensuring the
cache is genuinely filled to capacity.  This \emph{filled-context} methodology
produces qualitatively different results from the allocation-only baseline.

Context-length sensitivity is \textbf{non-monotonic with quantization level}---
mirroring the throughput ordering anomaly from RQ1.
Table~\ref{tab:ctx2048} summarises decode throughput from ctx=256 to ctx=2048.

The striking finding is that \textbf{Q2\_K, the fastest variant at short context,
is also the most fragile at long context}: decode throughput collapses from
7.97~tok/s (ctx=256) to 4.76~tok/s (ctx=2048), a $-40\%$ degradation, with
the cliff onset occurring as early as ctx=768 ($-21\%$).
By contrast, \textbf{Q3\_K\_M is the most context-stable variant}: throughput
holds at $5.01{\pm}0.27$~tok/s across the full 256--2048 token range
($<\pm5\%$ variation over an 8$\times$ context increase).
Q4\_K\_M similarly shows only $-6.6\%$ decline at ctx=2048 (5.57 to 5.21~tok/s).

Mid-precision variants exhibit moderate degradation: Q5\_K\_M loses $-26\%$
(4.46 to 3.31~tok/s, cliff at ctx=1200), Q8\_0 loses $-19\%$ (noisy, cliff at
ctx=1300), and Q4\_K\_S loses $-15\%$ (cliff at ctx=1200).
Q6\_K shows mild degradation of $-11\%$ (3.54 to 3.16~tok/s, cliff at ctx=1400).

This pattern is mechanistically informative.
The Llama~3.2~3B KV cache at 2048 tokens occupies $\approx$224~MB
(2 $\times$ 28 layers $\times$ 8 heads $\times$ 128 head-dim $\times$ 2048 $\times$ 2 bytes).
For Q2\_K, whose weight file is only 1.28~GB, this KV footprint represents
$\approx$17.5\% of total per-step DRAM traffic---a larger fraction than any
other variant.  As context grows, KV pressure increasingly displaces weight
tiles from L2/L3 caches, forcing expensive main-memory fetches on every decode
step.  Q3\_K\_M (1.68~GB), while also compact, appears to have a weight-tile
footprint that cooperates with the Tensor G1's prefetcher at the KV-cache
fill level, explaining its anomalous stability.

\textbf{Recommendation}: for deployments requiring ctx$\geq$1024, prefer
Q3\_K\_M (stable, 5.0~tok/s) or Q4\_K\_M (stable, 5.2~tok/s) over Q2\_K,
whose throughput advantage erodes above 768 tokens and fully disappears above
1400 tokens.  Q5\_K\_M and Q8\_0 should be treated as medium-context variants
(safe to ctx$\approx$1000).

Note: Flash Attention (\texttt{-fa} flag) was tested as a potential mitigation
for the Q2\_K cliff but was found to be unsupported by the \texttt{llama-completion}
binary build used in this study.  Evaluating FA-enabled builds is left for future work.

that BoolQ---which requires cross-sentence reasoning---is the more
informative quality discriminator among variants at this model scale.

\textbf{Instruction-following collapse at extreme quantization.}
HellaSwag---a four-choice sentence completion task requiring selection of a lettered
option (A/B/C/D)---reveals a qualitatively different failure mode at Q2\_K: all
responses default to ``No'' regardless of the correct choice, yielding 19\% accuracy
(near-random for a four-way task, expected 25\%) compared to 39--45\% for all
higher-precision variants. This collapse does not appear on BoolQ (yes/no format,
Q2\_K scores 69\%) or ARC-Easy (76\%, only 6 points below the best variant),
indicating a task-format-specific failure: at 3.40 bits per weight, the model
retains factual recall and binary judgment but loses the ability to follow
structured multi-choice sentence-completion prompts. Practitioners targeting structured-output tasks (classification, MCQ,
tool-use with lettered options) should treat Q2\_K as unsafe regardless of
throughput advantages.

Figure~\ref{fig:pareto} plots the Pareto frontier for all variants.
Q4\_K\_S (74\%, 4.8~tok/s) achieves the highest BoolQ accuracy of any variant,
while Q2\_K (69\%, 5.11~tok/s) leads on raw decode throughput.
Q4\_K\_M (72\%, 4.80~tok/s) provides a stable middle ground with the lowest
context-length sensitivity.
Q6\_K is strictly dominated on every metric: slower than Q4\_K\_M,
less accurate than Q4\_K\_S, and no more context-stable.
Notably, \textbf{Q8\_0 is no longer Pareto-optimal under BoolQ}: it scores 68\%
accuracy at 4.54~tokens/s, dominated by Q4\_K\_M which achieves both higher
accuracy \emph{and} higher throughput. Q3\_K\_M (69\%, 4.91~tokens/s) is also
dominated by Q4\_K\_M on BoolQ vs.\ throughput, though Q3\_K\_M leads on
context stability and TruthfulQA (see below). \textbf{Q6\_K (65\%, 3.52~tokens/s) is
strictly dominated on both dimensions}---simultaneously the lowest-scoring and
slowest practical variant.

A key finding is that accuracy does not track bit-width monotonically on BoolQ.
Q4\_K\_S (4-bit, small) achieves the \emph{highest} accuracy at 74\%;
Q4\_K\_M (4-bit, medium) follows at 72\%; Q6\_K (6-bit)
scores \emph{lowest} at 65\%---fewer correct answers than any other variant despite
having more bits per weight than either 4-bit variant.
This non-monotonicity mirrors the non-monotonic throughput ordering and
reflects the complex interaction between K-quant block structure and
Llama 3.2's layer-specific weight sensitivity.

\textbf{MMLU accuracy.}
A 100-question MMLU evaluation was conducted across all quantized variants.
Results: Q2\_K=42\%, Q3\_K\_M=48\%, Q4\_K\_S=49\%, Q4\_K\_M=47\%, Q5\_K\_M=50\%, Q6\_K=48\%, Q8\_0=47\%.
MMLU accuracy clusters tightly between 42--50\% across all variants
(95\% Wilson CI width $\approx\pm10$pp), making pairwise distinctions statistically
unreliable at $n{=}100$ questions. Q2\_K is detectably weaker (42\%), while
Q3\_K\_M through Q8\_0 are statistically indistinguishable on MMLU alone.
Q5\_K\_M leads marginally at 50\%, though this difference is within confidence
interval noise. These results confirm that MMLU at $n{=}100$ is insufficient to
rank quantization variants and that BoolQ is the more discriminative on-device
quality benchmark for this model.

\textbf{TruthfulQA accuracy.}
A 100-question TruthfulQA multiple-choice (MC1) evaluation was conducted across all
seven variants, measuring factual consistency and resistance to common misconceptions.
Results: Q3\_K\_M=68\%, Q5\_K\_M=65\%, Q4\_K\_M=60\%, Q6\_K=60\%, Q8\_0=58\%,
Q4\_K\_S=57\%, Q2\_K=50\%.
Three findings are noteworthy.
First, \textbf{Q3\_K\_M leads at 68\%}---the highest truthfulness score despite
being mid-range on BoolQ (69\%) and slower than Q2\_K. This is the same variant
that shows near-zero context sensitivity (Section~\ref{sec:rq2}), suggesting a
consistent pattern: Q3\_K\_M's block quantization structure may preserve factual
representations better than its bit-count implies.
Second, \textbf{Q2\_K collapses to 50\%}---near-chance for binary MC1---while
scoring 69\% on BoolQ and exhibiting the HellaSwag instruction-following collapse (19\%).
These three failure modes (HellaSwag format, TruthfulQA factual coherence, long-context
throughput) converge on Q2\_K as a variant with fundamental limitations beyond its
throughput advantage.
Third, the \textbf{non-monotonic quality ordering extends to TruthfulQA}: Q3\_K\_M (3.95~bpw) outperforms
Q4\_K\_S (4.85~bpw), Q8\_0 (8.5~bpw), and Q6\_K (6.59~bpw) on truthfulness.
This corroborates the BoolQ and MMLU findings that bits-per-weight is a poor predictor
of downstream task quality.

\textbf{Thread-count scaling.} We measured Q4\_K\_M decode throughput at 1, 2, 4,

2 threads: 4.1~tokens/s;

performance cores.

(72\%, 4.80~tokens/s) offers the most context-stable performance profile.
Q6\_K (2.7~GB) and Q8\_0 (3.4~GB) require $\geq$ 6~GB. Q8\_0 (68\%, 4.54~tokens/s)
is viable but dominated by Q4\_K\_M on both axes under BoolQ evaluation.
Q6\_K is the weakest operating point: at 65\% BoolQ accuracy and
only 3.52~tokens/s it is strictly dominated---prefer Q8\_0 or Q4\_K\_M on 6~GB
devices instead.
F16 (6.4~GB) is not viable on any consumer device $\leq$ 8~GB.

Figure~\ref{fig:latency_dist} shows latency distributions. Q3\_K\_M and Q6\_K
exhibit tighter distributions ($\sigma \leq 0.40$ tokens/s), indicating more
predictable behavior suitable for latency-SLA applications.

\subsection{The Non-Monotonic Throughput Anomaly}
\label{sec:nonmonotonic}

The most theoretically surprising result is that Q6\_K (6-bit, 2.7~GB) is slower
than both Q4\_K\_M (4-bit, 2.0~GB) and Q8\_0 (8-bit, 3.4~GB). On a naive
bandwidth model, higher bit-width means more bytes per weight and lower throughput.
The expected monotonic ordering would be Q2\_K $>$ Q3\_K\_M $>$ Q4\_K\_M $>$
Q6\_K $>$ Q8\_0. Our measured ordering is Q2\_K $>$ Q4\_K\_M $>$ Q8\_0 $\approx$
Q3\_K\_M $>$ Q6\_K.

We attribute the anomaly primarily to \textbf{kernel implementation overhead}
in the llama.cpp Q6\_K de-quantization path. The Q6\_K format packs 6-bit weights
using a combination of a 4-bit lookup table and 2-bit residuals, requiring
more complex de-packing logic per block compared to Q8\_0 (single 8-bit lookup)
or Q4\_K\_M (well-optimized 4-bit NEON kernel with vectorized scale application).
The de-quantization overhead partially offsets Q6\_K's memory bandwidth advantage,
yielding effective throughput below its bandwidth consumption would predict.

Similarly, Q3\_K\_M's moderate throughput (4.91~tokens/s despite 3-bit weights)
reflects the overhead of its medium-complexity block structure, which requires
scale factor lookups at finer granularity than Q4\_K\_M's implementation.

This finding highlights that the throughput ordering of quantization variants
is not determined by model size alone, but by the joint optimization of
memory bandwidth and kernel de-quantization efficiency. Practitioners should
treat Q6\_K with caution on ARM CPU backends.

\subsection{Memory Bandwidth as the Decode Bottleneck}

The context-length stability of Q4\_K\_M, Q2\_K, and Q8\_0 (Section~\ref{sec:rq2})
provides strong evidence that decode throughput on this device is
\textbf{memory-bandwidth saturated} by weight loading for these variants.
For Q4\_K\_M at 4.80~tokens/s, each decode step loads approximately 2.0~GB
of weights, implying an effective DRAM bandwidth utilization of
$2.0 \times 4.80 \approx 9.6$~GB/s for weight loading alone.
The KV-cache at 2048-token context contributes $\approx$224~MB per step,
$\approx$11.2\% of weight traffic---explaining Q4\_K\_M's near-zero throughput
variation across 256--2048 tokens. The Q3\_K\_M and Q6\_K collapses at ctx=2048
(Table~\ref{tab:ctx2048}) suggest that their quantization kernels incur higher
effective memory-access amplification that makes them sensitive to the larger
allocation footprint when the KV-cache grows to 224~MB alongside these models.

\subsection{Deployment Recommendations}

\textbf{Default recommendation: Q4\_K\_M.} At 4.80 tokens/s, 2.0~GB, and 72\%
BoolQ accuracy with low variance (std~0.52) and the lowest context-length sensitivity,
Q4\_K\_M provides the best overall operating point for 6~GB devices.
It simultaneously outperforms Q8\_0 and Q3\_K\_M on both accuracy and throughput.
Q4\_K\_M's performance stability and broad community ecosystem support further
reinforce it as the pragmatic default.

\textbf{Accuracy-critical deployments: Q4\_K\_S.} At 1.56~GB and 74\% BoolQ accuracy,
Q4\_K\_S achieves the highest task accuracy of any variant at negligible size cost vs.\ Q4\_K\_M.
It is the best choice for deployments where task accuracy is the primary constraint.

\textbf{Speed-critical deployments: Q2\_K.} At 5.11~tokens/s and only 1.3~GB,
Q2\_K is the fastest practical option (69\% BoolQ, speed-dominant).
It is the best choice when throughput matters most and RAM is limited to 2~GB,
\emph{provided} the target task does not involve structured multi-choice prompts
(see HellaSwag collapse finding in Section~\ref{sec:rq4}).

\textbf{Memory-constrained deployments (3--4~GB devices): Q2\_K or Q3\_K\_M.}
Both fit comfortably in 2~GB and can run on 3~GB devices. Q3\_K\_M is generally
preferred: it matches Q2\_K on BoolQ (both 69\%), leads on TruthfulQA (68\%),
and is context-stable through ctx=2048. Q2\_K is the right choice \emph{only}
when short-context throughput is the sole constraint and MCQ/truthfulness tasks are absent.

\textbf{Avoid Q6\_K on CPU backends.} Q6\_K is strictly dominated in both
BoolQ accuracy (65\% vs.\ 69--74\% for all others) and throughput (3.52~tokens/s,
the slowest practical variant). Q4\_K\_M is faster, smaller, and 7\% more accurate;
Q8\_0 is more accurate and nearly 29\% faster. This recommendation may not apply
to GPU-accelerated backends where the Q6\_K de-quantization overhead differs.

\textbf{Use 4 threads on heterogeneous ARM.} On the Tensor G1 (and similar
big.LITTLE ARM SoCs), 4 threads matches the performance-core count and delivers
peak throughput. Do not use all available threads.

\textbf{KV-cache window management (critical).} Context sensitivity is
\emph{non-monotonic} and does not follow bit-width ordering.
Under filled-context conditions, Q3\_K\_M ($<\pm5\%$) and Q4\_K\_M ($-7\%$)
are the safest choices for long-context deployments.
\textbf{Q2\_K is the worst choice for ctx$\geq$1024}: throughput degrades
$-40\%$ by ctx=2048, with the cliff beginning at ctx=768.
Q5\_K\_M ($-26\%$) and Q8\_0 ($-19\%$) are medium-context variants;
limit them to ctx$\leq$1024 for best performance.
Q6\_K shows mild degradation ($-11\%$) and Q4\_K\_S moderate ($-15\%$).
TTFT grows with KV-cache window size across all variants (due to larger cache
initialization cost), so responsiveness decisions should factor in TTFT budgets
alongside decode throughput.

\textbf{First}, Q4\_K\_S and Q4\_K\_M, which are mid-pack on ARM (rank 4),
leap to \emph{first and second} on Metal (19.9 and 19.2~tok/s).
Metal has highly tuned 4-bit dequantization kernels; the K-quant 4-bit
block format maps efficiently to GPU SIMD lanes.
\textbf{Second}, Q8\_0, which is mid-pack on ARM (rank 5, 4.54~tok/s),
is the \emph{slowest} variant on Metal (6.39~tok/s, rank 7).
The 8-bit format saturates GPU memory bandwidth without GPU-side
kernel advantages, making it the worst operating point on Metal
despite being a simple unstructured format.
\textbf{Third}, Q2\_K drops from the ARM champion (rank 1, 5.11~tok/s)
to third on Metal (17.79~tok/s)---still fast in absolute terms but
no longer dominant.

The throughput ratio between ARM and Metal (Q4\_K\_M: $19.22 / 4.80 = 4.0\times$)
is consistent with the GPU reference baseline gap reported in
Table~\ref{tab:gpu_baseline}, confirming that the Metal backend is
quantitatively comparable to a shared T4 GPU for this model.

These reversals confirm that the non-monotonic ARM ordering is
\emph{architecture-specific}: it is driven by ARM NEON dequantization
kernel overhead, which is absent on GPU backends with dedicated 4-bit
dispatch paths. Practitioners deploying on Metal (iOS, macOS) should
target Q4\_K\_S or Q4\_K\_M for maximum throughput; on ARM mobile,
Q2\_K leads for short-context throughput, but Q3\_K\_M is superior
for context stability, factual accuracy, and MCQ tasks.

% ===========================================================================
\section{Limitations and Future Work}
\label{sec:limitations}
\label{sec:future}
% ===========================================================================

\subsection{Current Limitations}

\textbf{Single-device evaluation.} All results are from a single Google Pixel 6a.
The Tensor G1's specific characteristics may not generalize to Snapdragon,
MediaTek, or Apple ARM SoC families.

\textbf{Quality evaluation scope.} The 100-question BoolQ and ARC-Easy evaluations
cover reading comprehension and elementary factual recall; broader coverage
(MMLU, GSM8K, HumanEval) would provide a more complete quality characterization.
WikiText-2 perplexity was measured on a 50K-token subset (${\approx}250$\,KB)
rather than the full 285K-token test set; while sufficient to establish the
PPL ordering and confirm the accuracy-proxy limitation, fine-grained PPL
comparisons may shift slightly with the full corpus.
The non-monotonic BoolQ accuracy ordering warrants further investigation.

\textbf{Working memory measurement only.} The peak memory figures in Figure~\ref{fig:memory}
reflect the MemAvailable delta in \texttt{/proc/meminfo}---i.e., the KV-cache plus
activation working memory---not the full RSS including memory-mapped model weights.
The model weights themselves contribute an additional 1.3--3.4~GB depending on
variant, loaded via mmap.

\textbf{Battery measurement granularity.} The benchmark was conducted over WiFi ADB
with the device unplugged, draining from 78\% to ${\sim}$2\%
(${\sim}$2.1\%/1k~tokens across ${\sim}$35{,}000 output tokens).
Android's \texttt{dumpsys battery} reports only integer-percentage values; each
30-second trial consumes $<$1\%, making per-trial tracking noise-dominated
(63 of 277 trials show a 1\% drop, the rest 0\%). Per-variant energy comparison
requires a hardware power monitor (e.g., Monsoon Solutions) or
longer-duration trials ($\geq$256 output tokens).

\textbf{Thermal trajectory not characterized.} Despite a 120-second cooldown
between experiments, cumulative thermal state over the 3-hour benchmark session
may have affected later results. We report results in run order but do not
model thermal drift explicitly.

\subsection{Future Work}

\textbf{Full-corpus perplexity and imatrix calibration.}
Re-evaluating PPL on the full WikiText-2 test set (285\,K tokens) will validate
fine-grained comparisons. On-device benchmarking of importance-matrix-calibrated
re-quantizations (\texttt{Q2\_K}--\texttt{Q8\_0}) will assess whether calibration
narrows the accuracy gap at lower bit-widths.

\textbf{Cross-platform comparison.} Repeating this benchmark on Snapdragon 8 Gen 3,
Apple A-series, and MediaTek Dimensity platforms would establish whether the
non-monotonic throughput finding is universal across ARM GGUF inference or
specific to the Tensor G1's memory subsystem.

\textbf{GPU and NPU acceleration.} The Tensor G1 includes an Arm Mali-G78 GPU
and a Google TPU block. Integrating these via OpenCL or delegating to TFLite's
NNAPI backend could dramatically change the efficiency--quality Pareto frontier.

\textbf{Context lengths beyond 2048.} We found that Q3\_K\_M and Q6\_K collapse
at ctx=2048. Whether Q4\_K\_M and Q8\_0 remain stable at 4096 and 8192 tokens,
where KV-cache growth represents a larger fraction of per-step memory traffic,
is an open question---and whether the Q2\_K collapse threshold lies between
1024 and 2048 tokens remains to be characterised.

\textbf{Alternative frameworks.} Evaluating MLC-LLM~\cite{mlcllm} and a
future stable ExecuTorch~\cite{executorch} release on the same hardware would
clarify whether the Q6\_K anomaly is llama.cpp-specific or intrinsic to ARM NEON.

\textbf{KV-cache quantization.} llama.cpp supports Q8\_0 quantization of the
KV cache via the \texttt{-ctk q8\_0 -ctv q8\_0} flags, halving the ${\approx}230$\,MB
KV footprint at 2048-token context. Measuring whether this enables the Q3\_K\_M and
Q6\_K collapse to be mitigated at ctx=2048---without accuracy regression---is an
immediately tractable experiment requiring no quantization re-runs.

\textbf{Speculative decoding with a draft model.} Meta's Llama 3.2 1B Instruct
provides a natural draft model for speculative decoding~\cite{leviathan_speculative}
with the Llama 3.2 3B verifier. Combined GGUF size (Q4\_K\_M 3B + 1B) is ${\approx}2.7$\,GB,
fitting within the Tensor G1's 6\,GB budget. At the memory-bandwidth-saturated
decode regime observed here, CPU speculative decoding may yield 1.2--1.8$\times$
speedup with a suitable calibration corpus for the draft model.

\textbf{P-core CPU affinity.} llama.cpp supports \texttt{--cpu-mask} to pin
inference threads to specific cores. Restricting to the 2$\times$ Cortex-X1 and
2$\times$ Cortex-A76 performance cores (mask \texttt{0x0F}) avoids the
synchronization penalty from little-core involvement observed at 8 threads, and
community measurements suggest 10--25\% further throughput improvement.

\textbf{Broader quality evaluation.} Adding GSM8K (grade-school math, 5-shot)
and MT-Bench instruction-following evaluation would characterize the accuracy
cliff below Q4\_K\_M on reasoning tasks, where perplexity gives little advance
warning of quality degradation~\cite{tang_eval}.

\section{Conclusion}
% ===========================================================================

We have presented a rigorous, multi-factor benchmark of seven GGUF quantization
variants of Llama 3.2 3B Instruct on a Google Pixel 6a (Tensor G1, 6~GB LPDDR5),
yielding 700+ valid measurements across throughput, context-scaling, and quality
benchmarks spanning four context lengths (256--2048 tokens) and a thread-count
scaling study.

Our primary empirical findings are:

\begin{enumerate}
  \item \textbf{Non-monotonic throughput}: Q6\_K (6-bit, 2.7~GB) delivers
        3.52~tokens/s---slower than Q4\_K\_M (4.80~tokens/s) and Q8\_0 (4.54~tokens/s)
        due to kernel de-quantization overhead on ARM NEON.
  \item \textbf{Context sensitivity is non-monotonic with bit-width}:
        Q3\_K\_M is the most context-stable variant ($<\pm5\%$ across 256--2048~tokens).
        Q4\_K\_M also shows low degradation ($-7\%$).
        Counterintuitively, Q2\_K---the fastest at short context---is the
        \emph{most} context-sensitive, losing $-40\%$ decode TPS from ctx=256 to 2048
        (cliff onset at ctx=768).  Q5\_K\_M ($-26\%$) and Q8\_0 ($-19\%$) show
        substantial degradation.  This non-monotonic pattern mirrors the throughput
        ordering anomaly and is explained by the relative KV-cache pressure on each
        variant's weight-tile footprint.
  \item \textbf{Thread-count sweet spot}: 4 threads is optimal on the Tensor G1's
        heterogeneous cluster; 8 threads degrades throughput by 20\% due to
        little-core load imbalance.
  \item \textbf{BoolQ accuracy non-monotonicity}: On-device BoolQ reading comprehension
        accuracy does not monotonically improve with bit-width (65--74\%):
        Q4\_K\_S (4-bit, small) achieves the highest score at 74\%, Q6\_K (6-bit)
        scores lowest at 65\% despite having more bits per weight.
        ARC-Easy factual accuracy ranges 76--82\% (Q4\_K\_M highest, Q2\_K lowest),
        non-monotonic with bit-width and narrower than BoolQ variance.
        Q2\_K exhibits instruction-following collapse on HellaSwag MCQ tasks (19\%),
        despite 69\% on BoolQ, revealing a task-format-specific failure mode.
        Q6\_K is strictly dominated on both BoolQ accuracy and throughput,
        confirming it as the weakest deployment choice.
  \item \textbf{Q4\_K\_S leads on accuracy; Q4\_K\_M recommended as default}:
        Q4\_K\_S (74\%, 4.8~tokens/s) achieves the highest BoolQ accuracy of any
        variant. Q4\_K\_M (72\%, 4.80~tokens/s) offers the most context-stable
        operating point and is the recommended default. Q2\_K (69\%, 5.11~tokens/s)
        is the speed-dominant choice for throughput-critical deployments, subject to
        the HellaSwag instruction-following collapse caveat.
        Q8\_0 is dominated by Q4\_K\_M on both accuracy and throughput axes.
        Q4\_K\_M (2.0~GB) offers the best overall balance for 6~GB deployment.
\end{enumerate}

These findings provide actionable guidance: use Q4\_K\_S when task accuracy is
paramount (highest BoolQ at 74\%); use Q4\_K\_M as the default
(context-stable, 72\% BoolQ, best overall balance); use Q2\_K for
latency-constrained or RAM-limited deployments (fastest at 5.11~tok/s,
69\% BoolQ, but unsafe for structured MCQ tasks); use 4 threads on big.LITTLE ARM;
and avoid Q6\_K on CPU backends entirely---it is strictly dominated on both
BoolQ accuracy and throughput, and catastrophically slow at ctx=2048. As on-device AI
continues to proliferate, hardware-specific benchmarks of this kind are essential
for evidence-based quantization selection.

\section{Experimental Reproducibility}

\subsection{Build Configuration}

The llama.cpp library was compiled from commit \texttt{b1-1a29907} using the
Android NDK version 29.0.14206865. Compilation flags:
\begin{verbatim}
-O3 -march=armv8.2-a+dotprod+fp16
-DGGML_USE_CPU_HBM=OFF
-DLLAMA_BUILD_TESTS=OFF
\end{verbatim}

The resulting \texttt{llama-cli} binary was pushed to \texttt{/data/local/tmp/}
on the device and executed via ADB shell. The Android app was compiled with
Android NDK 29.0.14206865, targeting arm64-v8a, using CMake 4.2.3 and Gradle 8.x.
All source code, benchmark scripts, and raw results are available in the
project repository.

\subsection{Invocation Template}

Each trial was executed with the following command template:
\begin{verbatim}
./llama-cli \
  -m <model_path>.gguf \
  -p "<system_prompt><user_prompt>" \
  -n 128 \
  --ctx-size <context_length> \
  --seed <trial_seed> \
  -t <thread_count> \
  --log-format json
\end{verbatim}

\subsection{Statistical Methodology}

Means and standard deviations in Table~\ref{tab:main_results} are computed
across 15 samples per variant per context (5 trials $\times$ 3 prompt types).
Warmup runs (2 per configuration) are excluded. Outlier rejection was not applied;
all completed trials within the timeout threshold are included in analysis.

\section{Quality Evaluation Details}

\subsection{BoolQ (100 questions)}
BoolQ questions are drawn from the SuperGLUE validation split~\cite{boolq}.
Each example consists of a passage (avg.\ $\sim$100 words) and a yes/no question.
The prompt template is:
\begin{verbatim}
Passage: {passage}
Question: {question}
Answer with only yes or no:
\end{verbatim}
Responses are scored by extracting the first occurrence of the words
\emph{yes} or \emph{no} (case-insensitive) from the model output,
within 400 characters following the instruction marker.

\subsection{ARC-Easy (100 questions)}
ARC-Easy questions are drawn from the AI2 ARC Easy test split~\cite{arc}.
Each example presents a 4-choice multiple-choice question.
The prompt template is:
\begin{verbatim}
Question: {question}
Choices: A) {A} B) {B} C) {C} D) {D}
Answer with only A, B, C, or D:
\end{verbatim}
Scoring uses exact-match on the predicted letter (A/B/C/D).
The \texttt{model\_output} field in stored JSON records the extracted letter only
(e.g., ``A''), not the raw llama-completion output, ensuring stored results are
directly verifiable and reproducible. Verified accuracy across all seven variants:
Q2\_K 76\%, Q3\_K\_M 78\%, Q4\_K\_S 81\%, Q4\_K\_M 82\%, Q5\_K\_M 81\%, Q6\_K 79\%, Q8\_0 80\%.
